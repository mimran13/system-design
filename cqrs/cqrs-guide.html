CQRS & Event Sourcing, from zero to mastery

A complete, code-first walkthrough built on a runnable NestJS bank. Read top to bottom and you'll understand not just how these patterns work, but why they exist and — just as importantly — when to walk away from them.

🏦 Bank-account domain 🧩 NestJS @nestjs/cqrs 🧪 Runs in-memory 📈 Staged: CQRS → +Event Sourcing

Every concept below points at a real file in src/. Keep the code open beside this page. The fastest way to learn this is to read a paragraph here, then go read the file it names — the files are commented to be read as a lesson, not just run.

🎯 How this guide is staged
We build the same bank twice. Stage 1 is pure CQRS: it stores the account balance directly, like a normal app, but splits reads from writes. Stage 2 keeps the CQRS split but throws away stored state entirely — it stores events and rebuilds state by replaying them. Doing it twice is the point: you'll feel exactly what Event Sourcing adds, and what it costs.

1Why CQRS exists

Start with the pain it solves. In a typical CRUD app there is one model — say an Account entity — and it does everything: it enforces business rules on writes, and it's also the thing you serialise onto a screen. That one model is pulled in two opposite directions:

  • Writes want a rich, rule-enforcing model. "You can't overdraw." "You can't ship a cancelled order." Writes care about invariants and consistency, and want a small, tightly-guarded object.
  • Reads want wide, denormalised, fast shapes. A dashboard wants an account plus its last 10 transactions plus a running total — often joined across many tables, shaped for one specific screen.

Serve both from one model and you get the classic mess: entities bloated with display fields, N+1 queries, read concerns leaking into your domain, caching bolted on awkwardly, and every change risking both paths at once.

CQRS — Command Query Responsibility Segregation — is the decision to stop compromising. Split the model in two. One model for changing state (the command side), a different model for answering questions (the query side). They can have different shapes, live in different stores, and scale independently.

🍽️ An analogy
A restaurant kitchen has an order ticket flow (commands: "table 4 wants the salmon") and a menu / specials board (queries: what diners read). You wouldn't force the chef to cook off the laminated menu, or ask diners to read the greasy order tickets. Same information, two representations, each shaped for its job. CQRS is that separation, in code.

That's the whole core idea. Everything else — buses, handlers, projections, event sourcing — is machinery in service of that one split. Hold onto it.

2The vocabulary

Six words carry the whole pattern. Learn them precisely — sloppiness here is why CQRS confuses people.

TermWhat it isGolden rule
CommandAn imperative message expressing intent: OpenAccount, Withdraw. Named as an order. May be rejected. Changes state. Exactly one handler.
QueryA request for data: GetAccount. Never changes state. No side effects. Exactly one handler.
HandlerThe code that processes one command or query type. Thin. Orchestrates; holds no business rules itself.
BusA dispatcher that routes a message to its one handler (CommandBus, QueryBus, EventBus). Decouples the sender from the handler. The caller never imports the handler.
EventA fact that already happened: MoneyWithdrawn. Past tense. Immutable. Cannot be rejected — it's history.
ProjectionAn event handler that updates a read model. The only writer to the read model. Turns events into query-ready shapes.
⚠️ The single trap
Command vs Event = intent vs fact. Withdraw (command) is a request that might be refused (insufficient funds). MoneyWithdrawn (event) is the record that it succeeded. One is future and rejectable; the other is past and permanent. Name commands as imperatives, events as past-tense. Mixing them up muddles everything downstream.

3The big picture

Here is the entire architecture on one diagram. Both stages of this repo follow it; only what sits inside "domain" and how it's persisted changes.

        ┌──────────────────────────── WRITE SIDE ────────────────────────────┐
 HTTP   │                                                                     │
 POST ─►│  Controller ─► CommandBus ─► CommandHandler ─► Domain model          │
        │   (thin)          (routes)         (orchestrates)    (enforces rules)│
        │                                          │                 │         │
        │                                          │            emits Event   │
        └──────────────────────────────────────────┼─────────────────┼────────┘
                                                    │                 ▼
                                              persists          ┌──────────┐
                                                    │           │ EventBus │
                                                    ▼           └────┬─────┘
                                            [ write store ]          │ publishes
                                                                     ▼
        ┌──────────────────────────── READ SIDE ─────────────────────────────┐
        │                                                    Projection        │
 HTTP   │  Controller ─► QueryBus ─► QueryHandler ─► [ read store ] ◄──updates│
 GET  ◄─│   (thin)         (routes)      (reads only)      (denormalised views)│
        └─────────────────────────────────────────────────────────────────────┘

Trace the two paths and notice they never share a model:

  • Write path (blue in the code): request → command → domain model → persistence → event. The domain model is the gatekeeper; nothing changes state without going through it.
  • Read path (purple): request → query → read store. It skips the domain entirely. It can't accidentally break a rule because it never touches the rule-holder.
  • The bridge: events flow from write to read via the EventBus, and a projection turns them into read-ready shapes. This is the only connection between the two sides.
🧭 In NestJS specifically
@nestjs/cqrs gives you CommandBus, QueryBus and EventBus out of the box. You write plain classes and tag them with @CommandHandler(X), @QueryHandler(X), @EventsHandler(X). Nest wires each handler to its message type automatically — you just list them as providers in the module. See src/stage1-cqrs/cqrs.module.ts.

4STAGE 1  Write vs read models

This is where CQRS earns its keep, so go slow. In Stage 1 we have two different objects representing an account, on purpose.

The write model — guardian of the rules

src/stage1-cqrs/domain/account.ts

export class Account {
  // No public setters. The ONLY way to change the balance
  // is through a method that enforces the rules.
  withdraw(amount: Money): void {
    if (amount.isZeroOrLess()) throw new InvalidAmountError();
    // the invariant, enforced at the only place it can break:
    if (amount.isGreaterThan(this.balance))
      throw new InsufficientFundsError(this.id);
    this.balance = this.balance.subtract(amount);
  }
}

Notice: no getters-and-setters data bag. The rule "an account may never go negative" lives inside the model, at the one place it could be violated. No controller, handler, or future teammate can forget it, because there's no other door into the balance. This is a rich domain model, and it's the heart of the write side. (The Money type it uses — src/shared/domain/money.ts — is a value object that stores integer pence so floating-point rounding can never lose you a penny.)

The read model — shaped for the screen

src/stage1-cqrs/read-model/account-view.ts

export interface AccountView {
  accountId: string;
  owner: string;
  balance: number;          // major units, ready to display
  transactionCount: number; // denormalised — write model never stores this
  lastActivityAt: string;
}

The read model looks nothing like the write model, and that's allowed — encouraged, even. It carries a display-friendly balance and a pre-computed transactionCount the write side never tracks. It has no methods and no rules, because its only job is to be read. In production this might live in a totally different database (Elasticsearch, a materialised view, a cache) tuned purely for reads.

Write model (Account)

  • Behaviour, not data
  • Enforces invariants
  • Loaded one at a time, mutated, saved
  • Optimised for correctness

Read model (AccountView)

  • Data, no behaviour
  • Denormalised for one screen
  • Queried in bulk, never mutated by queries
  • Optimised for fast reads

5STAGE 1  A command, end to end

Let's follow a single POST /stage1/accounts/:id/deposit through every layer. This is the write path made concrete.

 1. Controller     receives HTTP, validates the DTO, builds a command,
    (api/)         hands it to the CommandBus. Knows nothing else.
                        │
 2. CommandBus     routes DepositCommand ─► the one @CommandHandler(DepositCommand)
                        │
 3. Handler        loads the Account from the repository,
    (commands/)     calls account.deposit(...), saves it,
                    then publishes a MoneyDeposited event.
                        │
 4. Domain model   validates the amount, updates its balance. Rules live here.
                        │
 5. EventBus       carries MoneyDeposited ─► the projection
                        │
 6. Projection     updates the AccountView + appends a ledger row.
    (read-model/)   The read side is now current.

Step 1 — the controller is deliberately dumb

src/stage1-cqrs/api/accounts.controller.ts

@Post(':id/deposit')
async deposit(@Param('id') id: string, @Body() dto: AmountDto) {
  await this.commandBus.execute(new DepositCommand(id, dto.amountMinor));
  return { ok: true };
}

Look how little it does: validate input, build a message, dispatch it. It imports no repository, no domain object — only the bus. That's the segregation showing up right at the front door.

Steps 3–4 — the handler orchestrates, the model decides

src/stage1-cqrs/commands/deposit.command.ts

@CommandHandler(DepositCommand)
export class DepositHandler implements ICommandHandler<DepositCommand> {
  async execute(command: DepositCommand) {
    const account = await this.repository.findById(command.accountId);
    if (!account) throw new AccountNotFoundError(command.accountId);

    account.deposit(Money.fromMinor(command.amountMinor)); // ← model enforces rules
    await this.repository.save(account);

    this.eventBus.publish(/* MoneyDeposited */);        // ← tell the read side
  }
}
💡 The handler stays thin
The handler never checks "is there enough money?" — that's the model's job. It just loads, delegates, saves, announces. Keeping business logic in the model (not the handler) is what lets you unit-test the rules with no NestJS, no bus, no database. See src/stage1-cqrs/domain/account.spec.ts.

Step 6 — the projection bridges write to read

src/stage1-cqrs/read-model/account.projection.ts

@EventsHandler(MoneyDeposited)
export class MoneyDepositedProjection implements IEventHandler<MoneyDeposited> {
  handle(event: MoneyDeposited) {
    const view = this.store.getAccount(event.accountId);
    // recompute the denormalised view and append a ledger row
    this.store.upsertAccount({ ...view, balance: newBalance, ... });
    this.store.appendTransaction({ type: 'DEPOSIT', ... });
  }
}
⚠️ In Stage 1, events are just notifications
Critically: by the time we publish MoneyDeposited, the write model has already saved the new balance. The event exists only to let the read side catch up. If it were lost, the write model would still be correct — only the read model would drift. Remember this, because in Stage 2 the very same-looking event becomes the source of truth, and losing it would lose data. Same shape, completely different role.

6STAGE 1  The query path

The read path is refreshingly boring — which is the point. A query skips the domain model entirely and reads a pre-built shape.

src/stage1-cqrs/queries/get-account.query.ts

@QueryHandler(GetAccountQuery)
export class GetAccountHandler implements IQueryHandler<GetAccountQuery> {
  async execute(query: GetAccountQuery): Promise<AccountView> {
    const view = this.store.getAccount(query.accountId); // straight to read store
    if (!view) throw new AccountNotFoundError(query.accountId);
    return view;
  }
}

No Account is loaded. No rules run. There's nothing to compute — the projection already did that work when the event arrived. This is why CQRS reads are fast: the expensive shaping happens on write, once, not on every read. The "list my transactions" query is the clearest example — it serves a ledger the write model doesn't even store.

7Eventual consistency — the tax you pay

CQRS's power comes with a bill, and you must understand it before you use the pattern in anger.

The read model is updated after the write, by a separate step (the projection). In this demo the EventBus is synchronous, so the read model updates before the HTTP response returns — reads are instantly consistent. But in production, projections usually run asynchronously: events flow through a broker (Kafka, SQS, a queue), and the read model lags the write model by milliseconds to seconds.

  t0  POST /withdraw 30   ──► write model updated (balance now 120), event queued
  t1  200 OK returned to the client
  t2  ...event still in flight through the broker...
  t3  GET /account       ──► read model still says 150  ← STALE for a moment
  t4  projection catches up ──► read model now says 120  ← consistent again

That window between t1 and t4 is eventual consistency. It's not a bug; it's the deal. The consequences you must design for:

  • Read-your-own-writes: a user who just made a change may not see it immediately. Mitigate with UI tricks (optimistically show the new value), by reading from the write model for that one screen, or by versioned reads.
  • Never validate a command against the read model. The read model can be stale, so decisions (like "is there enough money?") must be made against the freshly-loaded write model — which is exactly what the handler does.
✅ What you buy for that tax
Independent scaling (throw read replicas at queries without touching writes), read models tuned per-use-case, and a write side kept clean of display concerns. If your app is read-heavy and the domain is complex, that's often a great trade. If it isn't — see §14 before you reach for this.

8STAGE 2  The leap: events as the source of truth

Everything so far was pure CQRS: we split reads from writes, but the write side still stored current state — a row that says balance = 150. Event Sourcing makes one radical change:

🔑 The one idea of Event Sourcing
Don't store the current state. Store the sequence of events that produced it, and derive state by replaying them. The event log is the database of record; the balance is just a running total you compute.

Instead of a row, an account is its history:

  CRUD / Stage 1 stores:          Event Sourcing / Stage 2 stores:
  ┌───────────────────┐           ┌────────────────────────────────┐
  │ balance = 150      │           │ #0 AccountOpened(100)           │
  └───────────────────┘           │ #1 MoneyDeposited(80)           │
   (the "after" photo)            │ #2 MoneyWithdrawn(30)           │
   history is lost                └────────────────────────────────┘
                                    balance = 100 + 80 − 30 = 150
                                    (replay the facts to get state)

Why on earth would you do this? Because the log is the feature:

  • Perfect audit trail. You don't bolt on history — history is all there is. Every change, in order, with its cause, forever. (Banks, healthcare, anything regulated loves this.)
  • Time travel. Replay the first N events and you know the exact state at any past moment. "What was the balance last Tuesday, and why?" is answerable. A CRUD system overwrote that.
  • Rebuildable read models. Because state is derived, you can invent a brand-new read model next year and back-fill it from day-one history (see §11). Read models become disposable.
  • Debugging & correction. Reproduce any bug by replaying the exact events that caused it. Fix a projection bug, replay, and the bad data is simply gone.

Try it live — this endpoint shows an account's raw truth:

$ curl localhost:3000/stage2/accounts/<id>/events
[ {"sequence":0,"type":"AccountOpened", ...},
  {"sequence":1,"type":"MoneyDeposited", ...},
  {"sequence":2,"type":"MoneyWithdrawn", "payload":{... "transfer":{...}}} ]

9STAGE 2  The aggregate: deciding vs applying

The event-sourced write model — the aggregate — is where the magic lives, and it has a beautiful split personality. Compare it to Stage 1's Account.withdraw(), which mutated state directly. The aggregate never mutates state directly. It has two kinds of methods:

src/stage2-event-sourcing/domain/account.aggregate.ts

export class AccountAggregate extends AggregateRoot {
  private balance: Money = Money.zero();
  public version = 0;

  // ── 1. COMMAND METHOD: decides WHETHER it's allowed, then emits an event.
  //        It validates against current state but does NOT change state.
  withdraw(amount: Money): void {
    if (amount.isGreaterThan(this.balance))
      throw new InsufficientFundsError(this.accountId);
    this.apply(new MoneyWithdrawn(this.accountId, amount.toMinor(), ...));
  }

  // ── 2. APPLY METHOD: the ONLY place state changes. Asks no questions —
  //        the event already happened, so it just folds it into state.
  //        NestJS calls on<EventName> automatically from apply()/replay.
  onMoneyWithdrawn(event: MoneyWithdrawn): void {
    this.balance = this.balance.subtract(Money.fromMinor(event.amountMinor));
    this.version++;
  }
}

Sit with why this split matters:

  • Command methods (withdraw) hold the rules. They run once, when the action first happens, and can reject it.
  • Apply methods (onMoneyWithdrawn) hold no rules — they trust the event as a done deal. They run every time the event is seen: once when it's created, and again on every future replay.
💡 Why validation must NOT be in the apply method
History is never re-judged. If onMoneyWithdrawn re-checked "enough funds?", then replaying an old, perfectly-valid withdrawal could throw years later just because the rule changed. Events are facts; you replay them, you don't re-litigate them. Decisions happen once, in the command method; folding happens forever, in the apply method.

NestJS's AggregateRoot.apply(event) does two things at once: it records the event as "uncommitted" (to be saved & published later) and immediately calls the matching on<Event> method so in-memory state stays live. Replaying history via loadFromHistory(events) calls the same on<Event> methods — but marks nothing uncommitted, because it's already saved.

10STAGE 2  The event store & loading by replay

There is no UPDATE accounts SET balance = ? anywhere in Stage 2. There's an append-only log and two operations against it.

Loading = read the stream + replay it

src/stage2-event-sourcing/infrastructure/account.repository.ts

async findById(id: string): Promise<AccountAggregate | null> {
  const stored = await this.eventStore.readStream(id); // every event, in order
  if (stored.length === 0) return null;

  const aggregate = this.publisher.mergeObjectContext(new AccountAggregate());
  aggregate.loadFromHistory(stored.map(e => this.deserialize(e))); // fold the facts
  return aggregate;
}

To load an account we fetch its entire event history and replay it into a fresh aggregate. The final in-memory balance is the sum of its facts. This is the cost of event sourcing — a load is a replay — and it's why real systems add snapshots (periodically save state and replay only events after it).

Saving = append the new events (with a concurrency check)

src/stage2-event-sourcing/infrastructure/event-store.ts

async append(streamId, events, expectedVersion) {
  const stream = this.streams.get(streamId) ?? [];
  // Optimistic concurrency: is the stream exactly where we left it?
  if (stream.length !== expectedVersion)
    throw new ConcurrencyError(streamId, expectedVersion, stream.length);
  // append-only — never update, never delete
  for (const event of events) stream.push(serialise(event));
}
🔒 Optimistic concurrency instead of locks
Two requests load the same account (both at version 3), both try to withdraw, both try to append expecting version 3. The first wins and moves the stream to 4; the second's "expected 3" no longer matches, so it's rejected with a ConcurrencyError — reload and retry. No row locks, no lost updates. The stream version is the concurrency token.

One honest detail worth reading in the code: events are stored as plain { type, payload } rows, so to replay them we must rebuild the right event class (replay keys off event.constructor.name). That's the deserialize() method — and in a real system it's where event versioning / upcasting lives, because events written years ago must still replay after their shape has evolved.

11STAGE 2  Replay & rebuilding projections

This is the payoff that makes event sourcing feel like a superpower. Because the read models are derived from the log, you can throw them away and rebuild them from scratch — any time, with a guarantee they'll be correct.

src/stage2-event-sourcing/projections/projection-rebuilder.ts

async rebuild() {
  this.readModel.reset();                       // wipe every view
  const events = await this.eventStore.readAll(); // the whole log, global order
  for (const stored of events)
    this.projector.project(deserialise(stored));  // same projector as live
  return { eventsReplayed: events.length };
}

The same projector code runs live (fed by the EventBus) and in bulk (fed by the log). That's what makes rebuilds boring and safe. See it work:

# do a few operations, note the balance, then:
$ curl -X POST localhost:3000/stage2/admin/rebuild-projections
{"eventsReplayed": 5}
# GET the account again → identical balance, reconstructed from nothing but the log

12STAGE 2  Sagas: coordinating across aggregates

A money transfer touches two accounts. But a core DDD rule says one command changes one aggregate — the aggregate is the consistency boundary, and you don't get an ACID transaction spanning two of them. So how do we move money from Alice to Bob?

With a saga (a.k.a. process manager): a long-running process built from single-aggregate steps, coordinated by events.

  TransferCommand(Alice→Bob, £70)
        │
        ▼
  [Alice] withdraw(£70, transferId) ──► MoneyWithdrawn { transfer: {to: Bob} }
                                              │
                                    TransferSaga sees it on the EventBus
                                              │
                                              ▼
                              DepositCommand(Bob, £70) ──► [Bob] MoneyDeposited

src/stage2-event-sourcing/sagas/transfer.saga.ts

@Saga()
moneyTransferred = (events$: Observable<IEvent>): Observable<ICommand> =>
  events$.pipe(
    ofType(MoneyWithdrawn),                 // only withdrawals…
    filter(e => !!e.transfer),               // …that belong to a transfer
    map(e => new DepositCommand(e.transfer.toAccountId, e.amountMinor, e.transfer)),
  );

A NestJS saga is a method that receives the stream of all events (an RxJS Observable) and returns a stream of commands; the framework dispatches them for you. Read it plainly: "whenever money is withdrawn as part of a transfer, deposit the same amount into the destination." Two separate transactions on two aggregates, stitched together by one rule.

⚠️ The honest caveat: compensation
Our saga implements the happy path. If the deposit failed (Bob's account was closed), money would have left Alice with nowhere to land. A production saga also listens for failure and emits a compensating command — here, deposit the £70 back into Alice to undo leg 1. That's "eventual consistency with compensation": the distributed-systems substitute for the single ACID transaction you can't have across aggregates. The code comments mark exactly where that logic goes.

13CQRS ≠ Event Sourcing (the confusion that trips everyone)

The single most common misunderstanding: people assume CQRS requires Event Sourcing, or that they're the same thing. They are independent. This repo proves it — Stage 1 is CQRS with zero event sourcing.

CQRSEvent Sourcing
Answers the questionHow do I structure reads vs writes?How do I store state?
Core moveTwo models: command side + query sideStore events, not current state
Can exist without the other?Yes (Stage 1)Yes — you can event-source without splitting reads (though it's rare)
Source of truthWhatever you store (a state row)The event log

They're combined so often because they fit like a glove: event sourcing makes your writes a stream of events, and CQRS gives your reads their own model so they don't pay the replay cost on every query. The events that ES produces are exactly what CQRS's projections want to consume. Great together — but choose them separately, for their own reasons.

14When to use it — and when to run away

This is the most important section for a senior engineer, and the one interviews probe hardest. These patterns are powerful and expensive. Reaching for them by default is a classic over-engineering trap.

✅ Reach for CQRS when

  • Read and write loads are wildly different and must scale separately
  • The read shapes differ a lot from the write model (many screens/reports)
  • The write side has genuinely complex domain rules worth isolating
  • You already tolerate (or want) eventual consistency

✅ Add Event Sourcing when

  • A full audit trail is a real requirement (finance, healthcare, legal)
  • You need temporal queries ("state as of…") or replay for debugging
  • The business naturally thinks in events (a ledger, a workflow)
  • You'll want to derive new read models from history later
🚫 Do NOT use these when…
  • It's a simple CRUD app. A boring Postgres table and a REST controller will out-ship and out-maintain any CQRS setup. Most apps are here.
  • Your team hasn't internalised eventual consistency — the bugs are subtle and land in production.
  • You'd apply it to the whole system. These patterns belong to specific bounded contexts with real complexity, not everywhere. Mixing a CQRS/ES core with plain CRUD elsewhere is normal and healthy.
  • You're doing it "to be scalable" with no measured need. That's the over-engineering smell.
🎙️ The interview-grade answer
"CQRS and Event Sourcing are separate tools. I'd apply CQRS to a bounded context where read and write concerns genuinely diverge and I can accept eventual consistency. I'd add Event Sourcing only where an audit log or temporal queries are real requirements — the ledger, say — and keep the rest of the system plain CRUD. The costs are eventual-consistency complexity, event versioning, and replay/snapshot management, so I don't reach for them by default." Saying when not to is what signals seniority.

17Test yourself

If you can answer these without peeking, you've got it. Click to reveal.

1. What is the single core idea of CQRS, in one sentence?

Use separate models for changing state (commands) and for reading state (queries), instead of forcing one model to do both. Everything else is machinery serving that split.

2. Command vs event — what's the precise difference?

A command is an imperative request that expresses intent and may be rejected (Withdraw). An event is a past-tense, immutable fact that already happened and cannot be rejected (MoneyWithdrawn). Intent vs record; future vs history.

3. In Stage 1, if a MoneyDeposited event were lost, is the balance wrong?

The write model is still correct — it saved the new balance before publishing. Only the read model would drift, because in pure CQRS the event is just a notification. In Stage 2 the answer flips: the event is the truth, so losing it loses data.

4. Why must validation live in the aggregate's command method, not its on<Event> method?

Because on<Event> runs on every replay, and history must never be re-judged. If it validated, replaying an old (valid) event could throw after the rules changed. Command methods decide once; apply methods fold facts forever without questioning them.

5. How does the event store prevent two concurrent withdrawals from both succeeding?

Optimistic concurrency. Each append states the expected stream version. Both loaded at version 3; the first append moves the stream to 4; the second still expects 3, no longer matches, and is rejected with a ConcurrencyError — reload and retry. No locks.

6. Why can you safely delete and rebuild the read model in Stage 2 but not (easily) in Stage 1?

In Stage 2 the read model is derived from the event log (the source of truth), so replaying the log reproduces it exactly. In Stage 1 the read model isn't derivable from anything more authoritative — the write state is just a row, with no history to replay — so a rebuild has nothing to rebuild from.

7. A transfer spans two accounts. Why not do it in one command handler with a transaction?

Because an aggregate is the consistency boundary and one command should change one aggregate; there's no ACID transaction across two. You use a saga: withdraw from the source (leg 1), and on that event fire a deposit command to the destination (leg 2), with a compensating refund if leg 2 fails. Eventual consistency with compensation.

8. You're asked to add CQRS + ES to a small internal CRUD tool. What do you say?

Push back. For simple CRUD, a plain table and REST controller ship faster and are easier to maintain. CQRS/ES pay off in specific bounded contexts with complex rules, divergent read/write loads, or real audit/temporal requirements — and they cost you eventual-consistency complexity, event versioning, and replay management. Knowing when not to use them is the senior signal.
