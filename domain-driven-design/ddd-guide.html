Chapter 1Introduction to DDD

Why does Domain-Driven Design exist, and when should you use it?

The Problem It Solves

In most projects, business logic ends up scattered. A UserService that does validation, database queries, email sending, and business calculations. Models that are just data bags with no behaviour. Rules duplicated across three different services. This is called an anemic domain model — the domain has no intelligence, it's just data.

DDD is a set of principles and patterns that puts the business domain at the centre of your code. Your code structure mirrors how the business thinks and talks.

Before DDD vs After DDD
❌ Without DDD (Anemic)
UserService
validate() + save() + sendEmail() + checkBalance() + applyRules()
User model
{ id, email, balance } — just data, no behaviour
✅ With DDD
BankAccount aggregate
deposit() withdraw() lock() — behaviour + rules
WithdrawMoneyUseCase
orchestrates only — load → act → save
EmailService (infrastructure)
technical concern, separate

The Two Pillars

🗺️ Strategic Design
The big picture. How to divide a large system into manageable parts with clear boundaries.
  • Ubiquitous Language
  • Subdomains
  • Bounded Contexts
  • Context Maps
🧱 Tactical Design
The building blocks. How to model business logic inside a single bounded context.
  • Entities, Value Objects
  • Aggregates, Repositories
  • Domain Events, Services
  • Factories, Specifications

When to Use DDD

Use DDD when…Skip DDD when…
Complex business rules that change frequentlySimple CRUD with no real business logic
Multiple teams working on the same systemSmall internal tools or admin panels
Business concepts are nuanced (finance, legal, medical)Data transformation pipelines
Long-lived product that will evolve for yearsPrototypes or throwaway code
✅ Rule of Thumb

DDD has a cost — it introduces layers and abstractions. Only pay that cost where the domain is complex enough to justify it. Use simpler patterns for the rest.

Chapter 2Ubiquitous Language ⭐

The most important DDD concept. The most often skipped.

🔑 Definition

A shared vocabulary between developers and domain experts where every term used in conversation is also the term used in the code. No translation. No mapping.

Why It Matters

Without a shared language, a business analyst says "redeem an investment", the developer hears it and writes processWithdrawal(), and six months later nobody can find the "redemption" feature in the codebase. Bugs happen because the translation between business language and code language introduces misunderstandings.

The Problem in Code

// ❌ Developer invented names — don't reflect business language
class UserService {
  processTransaction(userId: string, amt: number) { ... }
  handleAccountTermination(userId: string) { ... }
  doMoneyMovement(from: string, to: string, amount: number) { ... }
}

// ✅ Names come directly from domain expert conversations
class InvestmentAccount {
  redeemInvestment(amount: Money, reason: RedemptionReason): void { ... }
  closeAccount(closureReason: ClosureReason): void { ... }
}

class TransferService {
  transferBetweenAccounts(from: AccountId, to: AccountId, amount: Money): void { ... }
}

How to Build It

  • Event Storming — workshop with domain experts where you map out business events on sticky notes, revealing natural vocabulary
  • Domain Glossary — a shared document where each term is precisely defined and agreed upon
  • Code Reviews with domain experts — can they read your class names and understand what the system does?
  • Reject translations — if a dev says "user" but the business says "investor", use "investor" in the code
⚠️ Common Mistake

Using generic technical terms: UserManager, DataProcessor, Handler, Helper. These names come from the developer's world, not the business's world. If your domain expert can't understand your class names, your language is wrong.

💡 Key Insight

Ubiquitous Language is not just about naming. It's about alignment. When the code says account.redeemInvestment() and the business email says "the investor redeemed their investment", there's zero cognitive overhead mapping one to the other. The code is the business documentation.

Chapter 3Subdomains

Not all parts of your system deserve the same level of investment.

🔑 Definition

A subdomain is a distinct area of the business problem space. A large system is composed of multiple subdomains, each with different strategic value.

Three Types of Subdomains — Investment Platform Example
🌟 Core
Your competitive advantage. What makes you unique. Apply full DDD here.
Examples:
• Portfolio management
• Investment matching
• Risk calculation
• Redemption rules
🔧 Supporting
Necessary but not differentiating. Simpler design is fine.
Examples:
• User notifications
• Document storage
• Reporting
• KYC checks
📦 Generic
Commodity — buy off the shelf, don't build it.
Examples:
• Email delivery (SendGrid)
• Authentication (Auth0)
• PDF generation
• Payment gateway
✅ Rule of Thumb

Spend 80% of your DDD effort on the Core subdomain. For Supporting, use simpler patterns (Transaction Script, CRUD). For Generic, buy a SaaS product or use an open-source library. Don't build a better Stripe — integrate with it.

Chapter 4Bounded Contexts

The primary unit of strategic design. A boundary where a model is consistent.

🔑 Definition

A Bounded Context is an explicit boundary within which a particular domain model applies. The same word can mean completely different things in different bounded contexts.

The Key Insight: Same Word, Different Model

The word "Customer" in three different bounded contexts
Auth Context
Customer = login credentials

• id
• email
• passwordHash
• mfaEnabled
• lastLogin
Payments Context
Customer = billing entity

• id
• billingAddress
• paymentMethods[]
• taxId

KYC Context

• id

Each context has its own Customer class with its own fields, rules, and meaning. They share the same ID to reference each other, but are not the same model.

In NestJS

Each bounded context becomes a NestJS module (or a separate microservice). The module boundary IS the context boundary.

// src/
//   auth/             ← Auth bounded context (its own Customer)
//     domain/
//     application/
//     infrastructure/
//     presentation/
//     auth.module.ts
//
//   payments/         ← Payments bounded context (its own Customer)
//     domain/
//     ...
//     payments.module.ts
//
//   kyc/              ← KYC bounded context (its own Customer)
//     ...

// Contexts communicate through well-defined interfaces,
// NOT by importing each other's domain models directly.
// auth context needs to notify payments context:
@Injectable()
export class CustomerRegisteredHandler {
  @OnEvent('auth.customer.registered')
  async handle(event: CustomerRegisteredEvent) {
    // Creates a Payments-context Customer from the event data
    await this.paymentsService.createBillingProfile(event.customerId);
  }
}
⚠️ Common Mistake

Having one giant User model shared across the entire application with 40 fields. Every team adds their fields to it. It becomes a God Object that nobody fully understands. Instead, have multiple small, focused models — one per bounded context.

Chapter 5Context Maps

How bounded contexts communicate and relate to each other.

Once you have multiple bounded contexts, you need to define how they integrate. Context maps describe these relationships.

Anti-Corruption Layer (ACL) Most Important

When your context integrates with an external system (legacy system, 3rd party API), you build a translation layer that converts their model to your model. Your domain is never contaminated by external concepts.

// External payment provider has its own model
interface StripePaymentIntent {
  id: string;
  amount_received: number;    // in cents
  currency: string;
  payment_method_types: string[];
  status: 'succeeded' | 'processing' | 'failed';
}

// ✅ ACL: translate Stripe's model → your domain model
@Injectable()
export class StripePaymentAcl {
  translateToPaymentResult(stripeIntent: StripePaymentIntent): PaymentResult {
    return PaymentResult.create({
      externalId: stripeIntent.id,
      amount: Money.of(stripeIntent.amount_received / 100, stripeIntent.currency.toUpperCase()),
      status: this.mapStatus(stripeIntent.status),  // translate their status to yours
    });
  }

  private mapStatus(stripeStatus: string): PaymentStatus {
    const map = { 'succeeded': PaymentStatus.COMPLETED, 'processing': PaymentStatus.PENDING, 'failed': PaymentStatus.FAILED };
    return map[stripeStatus] ?? PaymentStatus.UNKNOWN;
  }
}

// Your use case only sees YOUR model — never Stripe's
class ProcessPaymentUseCase {
  async execute(command: ProcessPaymentCommand): Promise {
    const stripeResult = await this.stripeClient.createIntent(command.amount);
    const paymentResult = this.stripeAcl.translateToPaymentResult(stripeResult);  // ← ACL translates
    await this.paymentRepo.save(paymentResult);   // your domain, your model
  }
}

Context Map Relationship Patterns

These patterns describe the power dynamic and collaboration style between two bounded contexts. Choosing the right pattern is a team and architectural decision — it determines how much coupling you accept and who has the right to change the contract.

// Upstream = provides data/capability    Downstream = consumes it
[ BankAccount ] ──(upstream)──▶ [ FraudDetection ] ──(upstream)──▶ [ Notifications ]
                           (downstream)                (downstream)

1. Customer–Supplier Recommended for internal teams

The downstream (customer) has real influence over what the upstream (supplier) builds. Both teams negotiate the contract — upstream considers downstream's needs before shipping changes. This is the healthy default for two internal teams.

  • Upstream owns the API/events but consults downstream before breaking changes
  • Downstream usually adds an ACL to translate upstream's model to its own
  • Common in practice: BankAccount team (supplier) publishes events; FraudDetection team (customer) requests the fields it needs in those events
// BankAccount context (upstream / supplier)
// Publishes events — agrees with FraudDetection team on payload shape
export class MoneyWithdrawnIntegrationEvent {
  constructor(
    readonly accountId: string,
    readonly amount: number,
    readonly currency: string,
    readonly newBalance: number,   // ← FraudDetection asked for this field
    readonly occurredAt: string,
  ) {}
}

// FraudDetection context (downstream / customer)
// Has an ACL that maps supplier's event to its own model
@OnEvent('money.withdrawn')
async handle(event: MoneyWithdrawnIntegrationEvent): Promise {
  const withdrawal = this.acl.toFraudWithdrawal(event);  // ACL translates
  await this.fraudService.analyse(withdrawal);
}

2. Anti-Corruption Layer (ACL) Most Important

Already covered above — the downstream builds a translation layer so the upstream's model never leaks into its own domain. This is the implementation technique that pairs with Customer–Supplier or Conformist.

3. Open Host Service (OHS) + Published Language For shared infrastructure

Open Host Service: The upstream publishes a well-documented, stable public protocol that any number of consumers can use without negotiation. Unlike Customer–Supplier, the upstream does not customise for individual consumers.

Published Language: The formal, versioned schema of that protocol — usually OpenAPI, Avro, Protobuf, or JSON Schema. The published language IS the contract.

  • Upstream: "Here is our event schema v2. All consumers use it. We version it with breaking-change notices."
  • Downstream: No negotiation power — but the contract is explicit and stable
  • Example: Stripe's webhook events, an internal Identity service used by 10 teams
// Published Language: versioned event schema (the contract)
// bank-account-events.schema.v2.ts  ← shared schema package, not domain code
export interface MoneyWithdrawnEventV2 {
  version: 2;
  accountId: string;
  amount: number;
  currency: string;
  newBalance: number;
  occurredAt: string;
}

// Any downstream team consumes this — no negotiation needed
// If upstream needs to break the schema, they publish V3 alongside V2 (not replace it)

4. Conformist Use reluctantly

The downstream adopts the upstream's model as-is — no translation, no ACL. This creates tight coupling but is sometimes pragmatic when the upstream is a standard you can't influence.

  • Acceptable: ISO 4217 currency codes, country codes, HTTP status codes — these are standards; conforming is fine
  • Dangerous: Conforming to another team's internal domain model — you're now coupled to their refactoring decisions
// ✅ Conformist is fine when the upstream is an external standard
const currency = 'USD';  // ISO 4217 — we just adopt it, no translation needed

// ❌ Conformist is dangerous when the upstream is another team's domain model
import { BankAccount } from '../bank-account/domain/bank-account.aggregate';
// Now FraudDetection is directly coupled to BankAccount's internals.
// When BankAccount renames a field, FraudDetection breaks.

5. Shared Kernel Use sparingly

Two contexts share a small, explicitly agreed-upon piece of code (e.g., a common Money value object or a shared event schema package). Both teams must agree before changing it — this is the price of sharing.

  • Only share things that are truly stable and owned jointly (e.g., a @mycompany/shared-money npm package)
  • Never share domain logic — only share primitives, value objects, or event schemas
  • The shared code should be tiny. If it grows, it becomes a liability
// @mycompany/shared-kernel  (tiny shared npm package — both teams co-own)
export class Money {
  private constructor(readonly amount: number, readonly currency: string) {}
  static of(amount: number, currency: string): Money { ... }
  equals(other: Money): boolean { ... }
}

// BankAccount context and Payments context both use this Money
// But: if Payments needs Money to support crypto precision (8 decimal places),
// both teams must agree — Shared Kernel change requires coordination

6. Partnership For closely collaborating teams

Two teams evolve their contexts together in lockstep — both agree on changes simultaneously, both ship together. High coordination cost, but sometimes necessary during initial design when the boundary isn't stable yet.

  • Only viable when two teams work closely together and can release in sync
  • As the boundary stabilises, migrate to Customer–Supplier
  • In practice: two squads co-designing a new feature that spans two contexts

7. Separate Ways When integration isn't worth it

Two contexts have no integration at all — they solve similar problems independently. The cost of integration (translation, versioning, coordination) outweighs the benefit of sharing.

  • Example: a reporting context builds its own address formatting logic rather than calling the Customer context, because the coupling cost is too high for a low-value integration

Choosing the Right Pattern

SituationRecommended Pattern
Two internal teams, one provides data to the otherCustomer–Supplier + ACL
One service consumed by many teams (identity, payments)Open Host Service + Published Language
Integrating with external system / legacy codeACL (always)
Adopting an industry standard (ISO codes, HTTP)Conformist (acceptable)
Tiny, genuinely stable shared concept across two contextsShared Kernel (sparingly)
Two teams co-designing a new boundary togetherPartnership → migrate to Customer–Supplier
Integration cost > integration valueSeparate Ways
✅ Rule of Thumb

Default to Customer–Supplier + ACL for internal integrations and ACL alone for external systems. The ACL is your firewall — it keeps the upstream's model from leaking into your domain regardless of which relationship pattern you choose.

How Bounded Contexts Actually Talk to Each Other

The patterns above describe relationships. This section covers the mechanics — how data physically crosses the boundary at runtime.

Two Approaches

Synchronous (REST / gRPC)Asynchronous (Integration Events)
When You need real-time data to complete the current request You need to react to something that happened; stale-by-seconds is fine
Coupling Temporal — both services must be up at the same time Loose — publisher doesn't know or care who listens
Failure If downstream is down, your request fails Message broker buffers — downstream can be down and catch up later
DDD default Avoid unless truly necessary ✅ Preferred — keeps contexts truly independent
⚠️ The Most Common Mistake

Sharing a database between bounded contexts. Context A queries Context B's tables directly. This creates the tightest possible coupling — a schema change in B silently breaks A. Every context owns its data. Period.

The Async Event Flow (Most Common in DDD)

This is the full journey of data crossing a bounded context boundary via events:

// ── Step 1: Domain event raised inside the aggregate (BankAccount context) ──
// Inside bank-account.aggregate.ts
withdraw(amount: Money): Money {
  this._balance = this._balance.subtract(amount);
  this._domainEvents.push(
    new MoneyWithdrawnEvent(this._id.value, amount.amount, amount.currency, this._balance.amount)
  );
  return this._balance;
}

// ── Step 2: Use case publishes domain event AFTER save ──
// Inside withdraw-money.use-case.ts
await this.accountRepo.save(account);
for (const event of account.pullDomainEvents()) {
  this.eventEmitter.emit(event.constructor.name, event);  // in-process
}

// ── Step 3: Event handler translates domain event → integration event ──
// This lives in the application layer of the BankAccount context
@OnEvent('MoneyWithdrawnEvent')
async handleMoneyWithdrawn(event: MoneyWithdrawnEvent): Promise {
  // Domain event → Integration event (strip domain objects, use primitives only)
  const integrationEvent = {
    eventType: 'money.withdrawn',
    accountId: event.accountId,      // string — no domain objects cross the boundary
    amount: event.amount,            // number
    currency: event.currency,        // string
    newBalance: event.newBalance,    // number
    occurredAt: new Date().toISOString(),
  };

  // Publish to message broker (Kafka, RabbitMQ, SNS, etc.)
  await this.messageBroker.publish('bank-account.events', integrationEvent);
}

// ── Step 4: FraudDetection context (separate service) subscribes ──
// This lives in a completely different bounded context
@Injectable()
export class FraudDetectionEventConsumer {
  @MessagePattern('bank-account.events')
  async onBankAccountEvent(event: any): Promise {
    if (event.eventType !== 'money.withdrawn') return;

    // ✅ FraudDetection uses its OWN domain model — not BankAccount's
    const withdrawal = FraudWithdrawal.from({
      accountId: event.accountId,
      amount: event.amount,
      currency: event.currency,
    });

    await this.fraudAnalyser.analyse(withdrawal);
  }
}

The Golden Rules of Cross-Context Communication

RuleWhy
Never share domain objects across boundaries Integration events carry primitives only (strings, numbers). If you share Money from BankAccount into FraudDetection, you've coupled their models.
Never query another context's database Schema changes in one context silently break the other. Each context owns its data.
Store a local copy of what you need If FraudDetection needs account owner info, it stores its own copy (updated via events) — not a live JOIN to BankAccount's DB.
Translate at the boundary (ACL) Consumer maps the incoming event to its own domain model. A change in the publisher's event shape only requires updating the ACL, not rewriting domain logic.
Use the Outbox + Inbox pattern for guarantees Without Outbox, a crash between save and publish loses the event. Without Inbox, duplicate deliveries cause duplicate side effects. (See chapters 22 & 23.)

In a Modular Monolith (Our NestJS Setup)

You don't need a message broker to practice these patterns. In a single NestJS app with multiple modules, use NestJS EventEmitter2 as the in-process broker. Each module is a bounded context. The same rules apply — no direct imports of domain objects between modules.

// BankAccount module emits an integration-style event
this.eventEmitter.emit('money.withdrawn', {
  accountId: account.id.toString(),  // primitives only
  amount: amount.amount,
  currency: amount.currency,
  newBalance: account.balance.amount,
});

// FraudDetection module listens — it has ZERO imports from BankAccount module
@OnEvent('money.withdrawn')
async onMoneyWithdrawn(payload: MoneyWithdrawnPayload): Promise {
  // Works from payload primitives alone — no BankAccount domain classes needed
  await this.fraudService.checkWithdrawal(payload.accountId, payload.amount);
}

When you later extract FraudDetection to a microservice, you swap EventEmitter2 for Kafka/RabbitMQ. The domain logic inside FraudDetection doesn't change — only the transport layer does.

Chapter 6Value Objects

Immutable, self-validating objects defined entirely by their attributes.

🔑 Definition

A Value Object has no identity. Two Value Objects with the same attributes are considered equal. They are immutable — once created, never changed. Instead of modifying them, you create new ones.

The Three Rules

  • Immutable — no setters, no mutation after creation
  • Equality by value — Money(100, USD).equals(Money(100, USD)) is true
  • Self-validating — throws immediately if invalid, invalid state can never be created
export class Money {
  readonly amount: number;
  readonly currency: string;

  // Private constructor — the ONLY way to create is via factory methods
  private constructor(amount: number, currency: string) {
    if (amount < 0) throw new Error(`Money cannot be negative: ${amount}`);
    if (currency.trim().length !== 3) throw new Error(`Invalid currency: ${currency}`);
    this.amount = amount;
    this.currency = currency.toUpperCase();
  }

  static of(amount: number, currency: string): Money {
    return new Money(amount, currency);
  }

  static usd(amount: number): Money { return new Money(amount, 'USD'); }
  static zero(currency = 'USD'): Money { return new Money(0, currency); }

  // Operations return NEW instances — never mutate
  add(other: Money): Money {
    this.assertSameCurrency(other);
    return new Money(this.amount + other.amount, this.currency);
  }

  subtract(other: Money): Money {
    this.assertSameCurrency(other);
    if (other.amount > this.amount) throw new Error('Insufficient funds');
    return new Money(this.amount - other.amount, this.currency);
  }

  // Equality by VALUE, not by reference
  equals(other: Money): boolean {
    return this.amount === other.amount && this.currency === other.currency;
  }

  private assertSameCurrency(other: Money): void {
    if (this.currency !== other.currency)
      throw new Error(`Currency mismatch: ${this.currency} vs ${other.currency}`);
  }
}

// Usage
const price = Money.usd(100);
const tax   = Money.usd(8.5);
const total = price.add(tax);   // returns NEW Money(108.5, USD)
// price is still Money(100, USD) — immutable

// Two separate instances ARE equal
Money.usd(100).equals(Money.usd(100));  // ✅ true
Money.usd(100) === Money.usd(100);      // ❌ false (different references)

Other Common Value Objects

class Email {
  readonly value: string;
  private constructor(value: string) {
    if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value)) throw new Error(`Invalid email: ${value}`);
    this.value = value.toLowerCase();
  }
  static of(value: string): Email { return new Email(value); }
  equals(other: Email): boolean { return this.value === other.value; }
}

class AccountId {
  readonly value: string;
  private constructor(value: string) {
    if (!value?.trim()) throw new Error('AccountId cannot be empty');
    this.value = value;
  }
  static of(value: string): AccountId { return new AccountId(value); }
  static generate(): AccountId { return new AccountId(randomUUID()); }
}

class Percentage {
  readonly value: number;   // 0-100
  private constructor(value: number) {
    if (value < 0 || value > 100) throw new Error(`Invalid percentage: ${value}`);
    this.value = value;
  }
  static of(value: number): Percentage { return new Percentage(value); }
}
💡 Key Insight

Using AccountId instead of raw string makes your code self-documenting and prevents bugs. The TypeScript compiler will catch transactionRepo.findById(accountId) — wrong ID type! With raw strings, that bug is invisible at compile time.

Chapter 7Entities

Objects with a unique identity that persists over time.

🔑 Definition

An Entity has a unique identifier. Even if every attribute changes, it's still the same entity. Identity-based equality: two entities with the same ID are the same entity, regardless of their current attributes.

Entity vs Value Object

EntityValue Object
IdentityHas a unique IDNo identity, defined by attributes
EqualityBy IDBy attribute values
MutabilityAttributes can changeImmutable — replaced, not changed
LifetimeLong-lived, tracked over timeShort-lived, created as needed
ExamplesCustomer, Order, BankAccountMoney, Email, Address, Quantity
 
export class Customer {
  private readonly _id: CustomerId;
  private _email: Email;
  private _name: string;
  private _createdAt: Date;

  private constructor(props: { id: CustomerId; email: Email; name: string; createdAt: Date }) {
    this._id = props.id;
    this._email = props.email;
    this._name = props.name;
    this._createdAt = props.createdAt;
  }

  // Factory: creates a brand new customer
  static register(email: Email, name: string): Customer {
    if (!name.trim()) throw new Error('Customer name is required');
    return new Customer({ id: CustomerId.generate(), email, name, createdAt: new Date() });
  }

  // Factory: reconstitute from database (never generates a new ID)
  static reconstitute(props: { id: string; email: string; name: string; createdAt: Date }): Customer {
    return new Customer({ id: CustomerId.of(props.id), email: Email.of(props.email), name: props.name, createdAt: props.createdAt });
  }

  // Entities CAN change — but through controlled methods
  changeEmail(newEmail: Email): void {
    this._email = newEmail;
  }

  // Identity-based equality: same customer even if email changed
  equals(other: Customer): boolean {
    return this._id.value === other._id.value;
  }

  get id(): CustomerId { return this._id; }
  get email(): Email { return this._email; }
  get name(): string { return this._name; }
}

// Same customer before and after email change
const customer = Customer.register(Email.of('old@example.com'), 'Alice');
customer.changeEmail(Email.of('new@example.com'));
// customer.id is still the same — still the same entity

Chapter 8Aggregates & Aggregate Roots

Consistency boundaries that enforce business rules as a unit.

🔑 Definition

An Aggregate is a cluster of domain objects (entities + value objects) that are treated as a single unit for data changes. The Aggregate Root is the single entry point — external code ONLY interacts with the root, never directly with internal objects.

The Aggregate Root Rule

External code only touches the root
BankAccount Aggregate
BankAccount (Root) ← Only this is touched externally
_balance: Money (Value Object)
_isLocked: boolean
_version: number
✅ Correct
account.withdraw(amount)
❌ Wrong — bypasses rules
account['_balance'] = newValue
// All rules bypassed! No validation!

The Critical Insight: Keep Aggregates Small

A common mistake is loading everything into the aggregate. The aggregate should only contain what is strictly needed to enforce its business rules.

💡 Key Insight from the Bank Account Example

BankAccount's only rule is "balance cannot go negative". To enforce this, you only need the current balance. You do NOT need all 1000 transactions. Loading 1000 transactions every time someone deposits $10 is wasteful and doesn't scale.

What to include in aggregateWhat NOT to include
✅ Data needed to enforce business rules❌ Data only needed for display/reporting
✅ Status flags (isLocked, isActive)❌ Audit history (1000 transactions)
✅ Version for optimistic locking❌ Related entities from other aggregates
 
export class BankAccount {
  private readonly _id: AccountId;
  private _balance: Money;
  private _isLocked: boolean;
  private _version: number;       // for optimistic locking
  private readonly _createdAt: Date;

  // ✅ No _transactions array — not needed for business rules

  private constructor(props: { id: AccountId; balance: Money; isLocked: boolean; version: number; createdAt: Date }) {
    Object.assign(this, { _id: props.id, _balance: props.balance, _isLocked: props.isLocked, _version: props.version, _createdAt: props.createdAt });
  }

  // Factory: new account
  static open(currency = 'USD'): BankAccount {
    return new BankAccount({ id: AccountId.generate(), balance: Money.zero(currency), isLocked: false, version: 0, createdAt: new Date() });
  }

  // Factory: reconstitute from DB (used only by repository)
  static reconstitute(props: { id: string; balanceAmount: number; balanceCurrency: string; isLocked: boolean; version: number; createdAt: Date }): BankAccount {
    return new BankAccount({ id: AccountId.of(props.id), balance: Money.of(props.balanceAmount, props.balanceCurrency), isLocked: props.isLocked, version: props.version, createdAt: props.createdAt });
  }

  // Business method: enforces rule "can't withdraw from locked account"
  // and "balance can't go negative"
  withdraw(amount: Money): void {
    if (this._isLocked) throw new AccountLockedError(this._id.value);
    if (!this._balance.isGreaterThanOrEqual(amount)) throw new InsufficientFundsError(this._balance, amount);
    this._balance = this._balance.subtract(amount);
  }

  deposit(amount: Money): void {
    if (this._isLocked) throw new AccountLockedError(this._id.value);
    this._balance = this._balance.add(amount);
  }

  lock(): void { this._isLocked = true; }
  unlock(): void { this._isLocked = false; }
  incrementVersion(): void { this._version++; }

  get id(): AccountId { return this._id; }
  get balance(): Money { return this._balance; }
  get isLocked(): boolean { return this._isLocked; }
  get version(): number { return this._version; }
}

Chapter 10Domain Events ⭐

Not in original conversation  The mechanism for loose coupling between bounded contexts.

🔑 Definition

A Domain Event is something that happened in the domain, expressed as a past-tense fact. It's immutable, named in the past tense, and carries the data that describes what happened.

Examples: AccountOpened, MoneyDeposited, MoneyWithdrawn, AccountLocked, OrderPlaced, InvestmentRedeemed

Why Use Domain Events?

Without events, when money is withdrawn you might directly call fraudDetectionService.checkWithdrawal() from the use case. Now your BankAccount bounded context is coupled to the FraudDetection context. Add email notifications, analytics, audit logging — the use case becomes a growing list of direct calls.

With events, the use case just publishes MoneyWithdrawn. Every interested party subscribes independently. The bank account context doesn't know or care who's listening.

// 2. AGGREGATE raises events internally
export class BankAccount {
  private _domainEvents: object[] = [];

  withdraw(amount: Money): void {
    if (this._isLocked) throw new AccountLockedError(this._id.value);
    if (!this._balance.isGreaterThanOrEqual(amount)) throw new InsufficientFundsError(this._balance, amount);
    this._balance = this._balance.subtract(amount);
    // Raise event — NOT publish. The use case publishes AFTER saving.
    this._domainEvents.push(new MoneyWithdrawnEvent(this._id.value, amount.amount, amount.currency, TransactionId.generate().value));
  }

  // Drain events — use case calls this after saving
  pullDomainEvents(): object[] {
    const events = [...this._domainEvents];
    this._domainEvents = [];
    return events;
  }
}

// 3. USE CASE publishes events AFTER successful save
@Injectable()
export class WithdrawMoneyUseCase {
  constructor(
    @Inject(BANK_ACCOUNT_REPOSITORY) private readonly accountRepo: IBankAccountRepository,
    private readonly eventEmitter: EventEmitter2,    // NestJS EventEmitter2
  ) {}

  async execute(command: WithdrawMoneyCommand): Promise {

    account.withdraw(Money.of(command.amount, command.currency));

    await this.accountRepo.save(account);   // save first

    // THEN publish — if publish fails, account is still saved (acceptable)
    // If save fails, events are never published (correct — nothing happened)
    for (const event of account.pullDomainEvents()) {
      this.eventEmitter.emit(event.constructor.name, event);
    }
  }
}

Events are raised inside the aggregate (when the business action occurs), but published outside after a successful save. Never publish before saving — if the save fails, the event would describe something that never actually happened.

Not in original conversation  Encapsulate complex aggregate creation.

🔑 Definition

A Factory encapsulates the logic needed to create complex aggregates or value objects. When the creation process is non-trivial — multiple validation steps, loading related data, applying creation rules — extract it into a factory.

@Injectable()
export class BankAccountFactory {
  constructor(
    private readonly customerRepo: ICustomerRepository,
    private readonly accountLimitPolicy: AccountLimitPolicy,
  ) {}

  async openForCustomer(customerId: CustomerId, currency: string): Promise {

// Base class
abstract class Specification {
  abstract isSatisfiedBy(candidate: T): boolean;

  and(other: Specification): Specification {
    return new AndSpecification(this, other);
  }
  or(other: Specification): Specification {
    return new OrSpecification(this, other);
  }
  not(): Specification {
    return new NotSpecification(this);
  }
}

class AndSpecification extends Specification {
  constructor(private left: Specification, private right: Specification) { super(); }
  isSatisfiedBy(candidate: T): boolean {
    return this.left.isSatisfiedBy(candidate) && this.right.isSatisfiedBy(candidate);
  }
}

// Individual specifications — each encapsulates ONE rule
class AccountNotLockedSpec extends Specification {
  isSatisfiedBy(account: BankAccount): boolean { return !account.isLocked; }
}

class HasSufficientFundsSpec extends Specification {
  constructor(private required: Money) { super(); }
  isSatisfiedBy(account: BankAccount): boolean {
    return account.balance.isGreaterThanOrEqual(this.required);
  }
}

class AccountNotUnderReviewSpec extends Specification {
  isSatisfiedBy(account: BankAccount): boolean { return !account.isUnderReview; }
}

// Compose them
const canWithdraw = new AccountNotLockedSpec()
  .and(new HasSufficientFundsSpec(amount))
  .and(new AccountNotUnderReviewSpec());

if (!canWithdraw.isSatisfiedBy(account)) {
  throw new WithdrawalNotAllowedError(account.id.value);
}

// ✅ Testable — each spec is independently testable

Chapter 16Optimistic Locking

How to handle two requests modifying the same aggregate simultaneously.

The Problem: Lost Updates

Race condition — 50ms apart
❌ Without version check
Request 1: load account (balance=1000)
Request 2: load account (balance=1000)
Request 1: withdraw 200 → balance=800, SAVE ✅
Request 2: deposit 500 → balance=1500, SAVE ✅
Overwrites Request 1's withdrawal!
Correct balance should be 1300
✅ With version check
Request 1: load account (v=5, balance=1000)
Request 2: load account (v=5, balance=1000)
Request 1: save WHERE version=5 → v becomes 6 ✅
Request 2: save WHERE version=5 → 0 rows! version is 6 now
ConcurrentModificationError → retry
// AGGREGATE: carries a version number
export class BankAccount {
  private _version: number;
  incrementVersion(): void { this._version++; }
  get version(): number { return this._version; }
}

// ORM ENTITY: @VersionColumn() tells TypeORM to auto-manage version
@Entity('bank_accounts')
export class BankAccountOrmEntity {
  @VersionColumn()
  version: number;  // TypeORM auto-increments and adds WHERE version=N to updates
}

// REPOSITORY: TypeORM throws OptimisticLockVersionMismatchError on conflict
async save(account: BankAccount): Promise {
  try {
    await this.ormRepo.save(entity);
    account.incrementVersion();
  } catch (error) {
    if (error.name === 'OptimisticLockVersionMismatchError') {
      throw new ConcurrentModificationError(account.id.value);
    }
    throw error;
  }
}

// USE CASE: retry with exponential backoff
async execute(command: WithdrawMoneyCommand): Promise {
  for (let attempt = 1; attempt <= 3; attempt++) {
    try {
      return await this.tryWithdraw(command);
    } catch (error) {
      if (error instanceof ConcurrentModificationError && attempt < 3) {
        await new Promise(r => setTimeout(r, 50 * attempt));  // backoff
        continue;
      }
      throw error;  // non-concurrent errors: don't retry
    }
  }
}
// After 3 retries, throws 409 Conflict to the client

Pessimistic Locking — The Alternative

Optimistic locking assumes conflicts are rare and deals with them after the fact (retry). Pessimistic locking assumes conflicts are likely and prevents them upfront by locking the row at read time — no other transaction can read or write it until you're done.

Optimistic LockingPessimistic Locking
MechanismVersion number checked on writeDatabase row lock on read (SELECT FOR UPDATE)
Conflict handlingDetected at save time → retryPrevented at read time → second reader waits or times out
ThroughputHigh — no blockingLower — readers queue behind the lock holder
RiskWasted work on retry under high contentionDeadlocks if two transactions lock rows in opposite order
Best forLow-to-medium contention (most web apps)High contention on the same row (e.g., shared inventory counter)
 
// Pessimistic locking with TypeORM — locks the row for the duration of the transaction
const account = await this.repo.findOne({
  where: { id: accountId },
  lock: { mode: 'pessimistic_write' },  // issues SELECT ... FOR UPDATE
});
// No other transaction can read or write this row until we commit
account.withdraw(amount);
await this.repo.save(account);
// Lock released on commit
✅ When to Choose Which

Default to optimistic locking — it scales better and avoids deadlocks. Switch to pessimistic locking only when your retry rate is unacceptably high under load, indicating genuine high contention on the same aggregate instance. Most bank account operations don't need pessimistic locking — a single account rarely receives concurrent writes.

Chapter 18Event Sourcing

Not in original conversation  Store events, not state.

🔑 Definition

Instead of storing the current state of an aggregate, you store every event that ever happened to it. Current state is rebuilt by replaying the event log from the beginning.

Traditional vs Event Sourcing

Traditional (store state)
bank_accounts table:
id: acc_123
balance: 300 USD
← No history. What happened? No idea.
Event Sourcing (store events)
events table:
1. AccountOpened (balance=0)
2. MoneyDeposited (amount=500)
3. MoneyWithdrawn (amount=200)
Replay → current balance = 300 
// Events are the source of truth
type BankAccountEvent = AccountOpenedEvent | MoneyDepositedEvent | MoneyWithdrawnEvent;

class EventSourcedBankAccount {
  private _balance: Money = Money.zero();
  private _isLocked: boolean = false;

  // Reconstitute by replaying events
  static reconstitute(events: BankAccountEvent[]): EventSourcedBankAccount {
    const account = new EventSourcedBankAccount();
    for (const event of events) {
      account.apply(event);   // rebuild state from each event
    }
    return account;
  }

  // Apply: update internal state from event
  private apply(event: BankAccountEvent): void {
    if (event instanceof AccountOpenedEvent) {
      this._balance = Money.zero(event.currency);
    } else if (event instanceof MoneyDepositedEvent) {
      this._balance = this._balance.add(Money.of(event.amount, event.currency));
    } else if (event instanceof MoneyWithdrawnEvent) {
      this._balance = this._balance.subtract(Money.of(event.amount, event.currency));
    }
  }
}

Domain Events ≠ Event Sourcing. You can use Domain Events (Chapter 10) without Event Sourcing. Domain Events are about communication between contexts. Event Sourcing is a persistence strategy. They complement each other but are independent choices.

Chapter 19bThe Inbox Pattern

The consumer-side complement to the Outbox Pattern. Guarantees reliable, idempotent event processing.

🔑 Definition

The Inbox Pattern ensures that when a service receives an integration event, it is processed exactly once — even if the message broker delivers it multiple times. The incoming event is first persisted to an inbox table, then processed. If the service crashes mid-processing, the inbox retries from a known state.

Why Inbox Is Needed

Message brokers guarantee at-least-once delivery — the same message can arrive more than once (network retries, broker restarts). Without an Inbox, your handler runs the same event twice:

// ❌ Without Inbox — no protection against duplicate delivery
@OnEvent('MoneyWithdrawnIntegrationEvent')
async handle(event: MoneyWithdrawnIntegrationEvent) {
  // If broker delivers this twice → email sent twice, fraud check runs twice
  await this.emailService.sendWithdrawalAlert(event.accountId, event.amount);
  await this.fraudService.check(event.accountId, event.amount);
}

The Implementation

// 1. INBOX TABLE entity — persists incoming events before processing
@Entity('inbox_events')
export class InboxEventEntity {
  @PrimaryColumn('uuid')
  id: string;           // the event's own ID (from the publisher)

  @Column()
  eventType: string;

  @Column('jsonb')
  payload: object;

  @Column({ type: 'enum', enum: ['PENDING', 'PROCESSED', 'FAILED'], default: 'PENDING' })
  status: 'PENDING' | 'PROCESSED' | 'FAILED';

  @Column({ nullable: true })
  errorMessage: string;

  @CreateDateColumn()
  receivedAt: Date;

  @Column({ nullable: true })
  processedAt: Date;
}

// 2. MESSAGE CONSUMER — receives from broker, writes to inbox (does NOT process yet)
@Injectable()
export class KafkaConsumer {
  @EventPattern('MoneyWithdrawnIntegrationEvent')
  async receive(@Payload() event: MoneyWithdrawnIntegrationEvent) {
    // Check if already received (duplicate delivery guard)
    const existing = await this.inboxRepo.findOne({ where: { id: event.eventId } });
    if (existing) return;  // already in inbox — skip silently

    // Persist to inbox — if this crashes, broker will redeliver and we try again
    await this.inboxRepo.save({
      id: event.eventId,       // use the publisher's event ID as the inbox ID
      eventType: 'MoneyWithdrawnIntegrationEvent',
      payload: event,
      status: 'PENDING',
    });
    // Acknowledge message to broker — we've safely stored it
  }
}

// 3. INBOX PROCESSOR — background job that processes PENDING inbox events
@Injectable()
export class InboxProcessorJob {
  @Cron('*/3 * * * * *')  // every 3 seconds
  async processPending(): Promise {
    const pending = await this.inboxRepo.find({
      where: { status: 'PENDING' },
      order: { receivedAt: 'ASC' },
      take: 50,
    });

    for (const inboxEvent of pending) {
      try {
        await this.processEvent(inboxEvent);
        inboxEvent.status = 'PROCESSED';
        inboxEvent.processedAt = new Date();
      } catch (error) {
        inboxEvent.status = 'FAILED';
        inboxEvent.errorMessage = error.message;
        // Failed events can be retried, alerted, or moved to dead-letter
      }
      await this.inboxRepo.save(inboxEvent);
    }
  }

  private async processEvent(inboxEvent: InboxEventEntity): Promise {
    switch (inboxEvent.eventType) {
      case 'MoneyWithdrawnIntegrationEvent':
        const event = inboxEvent.payload as MoneyWithdrawnIntegrationEvent;
        await this.emailService.sendWithdrawalAlert(event.accountId, event.amount);
        await this.fraudService.check(event.accountId, event.amount);
        break;
      // ... other event types
    }
  }
}

✅ Rule of Thumb

Outbox + Inbox together give you end-to-end reliable messaging without distributed transactions. Outbox = "I will publish this exactly once." Inbox = "I will process this exactly once." Together they eliminate the two hardest failure modes in event-driven systems.

💡 Outbox vs Inbox — When Do You Need Each?

Outbox — needed on any service that publishes integration events. Prevents "event lost on publish failure".
Inbox — needed on any service that consumes integration events and performs non-idempotent operations (sending emails, charging cards, updating balances). If your handler is naturally idempotent, Inbox is optional but still good for observability.

The discovery technique for mapping complex domains — before writing any code.

🔑 Definition

Event Storming is a collaborative workshop format where developers and domain experts map out a complex business domain using sticky notes on a wall. It's the primary technique for discovering your bounded contexts, aggregates, and ubiquitous language.

Invented by Alberto Brandolini. The idea: get every stakeholder in a room and work backwards from business events. No diagrams, no UML — just sticky notes and conversation.

The Process

  1. Chaotic Exploration — everyone puts orange stickies (domain events) on the wall independently. No order, no structure yet. Just brainstorm what happens in the business.
  2. Enforce the Timeline — arrange events left-to-right in chronological order. Conflicts surface immediately — this is valuable.
  3. Add Commands — for each event, add the blue command that caused it. "Who decided this should happen?"
  4. Add Actors & Policies — who issues each command? What automatic reactions happen after each event?
  5. Add Aggregates — group commands and events around the aggregate that handles them.
  6. Identify Bounded Contexts — look for natural clusters of related aggregates. Naming conflicts (same word means different things in two clusters) reveal context boundaries.

What You Get Out of It

  • Your Ubiquitous Language — the domain expert's exact vocabulary is on the wall
  • Your Domain Events — already named, in past tense, agreed upon
  • Your Aggregates — naturally emerge from grouping commands + events
  • Your Bounded Contexts — where language breaks down, context boundaries exist
  • Your Hotspots — the red stickies show where domain knowledge is unclear or contested
✅ Rule of Thumb

Run an Event Storming session before writing any code in a new domain. Even 4 hours with the right people will save weeks of rework caused by modelling the wrong thing. The output maps directly to your DDD building blocks.

💡 Key Insight

The most valuable moment in Event Storming is when two people disagree on the name of an event or what an aggregate is called. That disagreement is a signal — either you have a language problem (fix with ubiquitous language) or you have a bounded context boundary (the same concept means different things to different people).

Chapter 24Integration Events vs Domain Events

Two types of events — same concept, very different scope and concerns.

⚠️ Common Confusion

Many developers use "domain event" and "integration event" interchangeably. They are fundamentally different in scope, delivery guarantees, and versioning concerns.

The Distinction

Domain Event
  • Within a single bounded context
  • In-process (same NestJS module)
  • Synchronous or async (EventEmitter2)
  • No versioning concerns (same codebase)
  • If handler fails, can be part of same transaction
  • Rich domain objects as payload
Integration Event
  • Crosses bounded context boundaries
  • Out-of-process (Kafka, RabbitMQ, SQS)
  • Always async, always eventual
  • Versioning is critical (consumers may be on old version)
  • Requires Outbox Pattern for reliability
  • Flat, serialisable DTO as payload (no domain objects)
// ── DOMAIN EVENT: raised inside BankAccount aggregate ──
// Lives in domain layer, rich domain types, in-process only
export class MoneyWithdrawnDomainEvent {
  constructor(
    readonly accountId: AccountId,   // rich domain type
    readonly amount: Money,           // rich domain type
    readonly occurredAt: Date = new Date(),
  ) {}
}

// Handled by another service in the SAME bounded context
@OnEvent('MoneyWithdrawnDomainEvent')
async handleFraudCheck(event: MoneyWithdrawnDomainEvent) {
  // Same process, same deployment, no serialisation needed
  await this.fraudService.check(event.accountId, event.amount);
}

// ── INTEGRATION EVENT: crosses bounded context boundary ──
// Lives in shared contracts / application layer
// Plain serialisable DTO — no domain objects
export class MoneyWithdrawnIntegrationEvent {
  readonly version = '1.0';   // versioning is essential
  constructor(
    readonly accountId: string,     // primitive, not AccountId VO
    readonly amount: number,         // primitive, not Money VO
    readonly currency: string,
    readonly occurredAt: string,     // ISO string, not Date (serialisable)
    readonly eventId: string,        // for idempotency on consumer side
  ) {}
}

// Published to message broker AFTER domain event is handled and saved
// (via Outbox Pattern — see Chapter 19)
@OnEvent('MoneyWithdrawnDomainEvent')

  const integrationEvent = new MoneyWithdrawnIntegrationEvent(
    domainEvent.accountId.value,         // convert to primitive
    domainEvent.amount.amount,
    domainEvent.amount.currency,
    domainEvent.occurredAt.toISOString(),
    randomUUID(),
  );
  // Write to outbox table (same transaction as aggregate save)
  await this.outboxRepo.save(integrationEvent);
}

// Must be idempotent — same event may arrive more than once

  // Check idempotency first
  if (await this.processedEvents.contains(event.eventId)) return;

  await this.emailService.sendWithdrawalAlert(event.accountId, event.amount);
  await this.processedEvents.markProcessed(event.eventId);
}

Event Schema Versioning

Integration events are a public contract. Once consumers are running in production, you can't just change the shape of an event — that breaks them. Versioning is how you evolve events safely.

Safe Changes (Non-Breaking)

These can be deployed without coordinating with consumers:

  • Adding a new optional field — consumers that don't know about it simply ignore it
  • Adding a new event type — consumers that don't subscribe to it are unaffected
// V1 — original
export interface MoneyWithdrawnEventV1 {
  version: 1;
  accountId: string;
  amount: number;
  currency: string;
}

// ✅ Safe: adding optional field — old consumers ignore it
export interface MoneyWithdrawnEventV1 {
  version: 1;
  accountId: string;
  amount: number;
  currency: string;
  newBalance?: number;  // optional — old consumers unaffected
}

Breaking Changes (Require a Strategy)

Renaming a field, changing a type, or removing a field will break existing consumers. The safe approach is dual publishing: publish both V1 and V2 simultaneously, let consumers migrate, then retire V1.

// ❌ Breaking: renaming a field
// V1: { amount: number }  → V2: { withdrawalAmount: number }
// Any consumer reading `amount` now gets undefined

// ✅ Dual publishing strategy
@OnEvent('MoneyWithdrawnDomainEvent')
async publishIntegrationEvent(domainEvent: MoneyWithdrawnDomainEvent) {
  // Publish V1 (for existing consumers still on old version)
  await this.outbox.save({
    topic: 'bank-account.events',
    payload: {
      version: 1,
      accountId: domainEvent.accountId.value,
      amount: domainEvent.amount.amount,       // old field name
      currency: domainEvent.amount.currency,
    },
  });

  // Publish V2 (for consumers that have migrated)
  await this.outbox.save({
    topic: 'bank-account.events.v2',           // or same topic + version filter
    payload: {
      version: 2,
      accountId: domainEvent.accountId.value,
      withdrawalAmount: domainEvent.amount.amount,  // new field name
      currency: domainEvent.amount.currency,
      newBalance: domainEvent.newBalance.amount,     // new field
    },
  });
}
// Once all consumers are on V2, retire the V1 publish block

Chapter 25Eventual Consistency

The trade-off you accept when bounded contexts communicate via events.

🔑 Definition

Eventual consistency means that after a change in one bounded context, other contexts will eventually reflect that change — but not immediately. There is a window where contexts are temporarily out of sync.

Strong Consistency vs Eventual Consistency

Strong Consistency
Everything updated in one database transaction. All readers see the change immediately. Limited to a single aggregate, single database.
BankAccount aggregate:
balance deducted + transaction recorded
→ one DB transaction
→ always consistent ✅
Eventual Consistency
Change published as an event. Other contexts react asynchronously. There's a delay (milliseconds to seconds) before everything is in sync.
BankAccount saved ✅
→ MoneyWithdrawn event published
→ Notifications context processes it (50ms later)
→ Email sent ✅ (eventually)

Where Each Applies

ScenarioConsistency ModelWhy
Balance deducted + transaction recordedStrongBoth are in the same aggregate / same DB transaction. Must be atomic.
Withdrawal confirmed + fraud check runsEventualFraud detection is a separate bounded context. Acceptable delay.
Withdrawal confirmed + email sentEventualEmail is infrastructure. A 1-second delay is fine.
Account balance + account summary viewEventualCQRS read model updated asynchronously. Stale by milliseconds.

Designing for Eventual Consistency

  • Make consumers idempotent — the same event may arrive more than once. Processing it twice must be safe.
  • Use event IDs for deduplication — store processed event IDs and skip duplicates.
  • Design UIs for it — show "your withdrawal is being processed" rather than an immediate balance update that might be wrong.
  • Define acceptable staleness — how stale can the read model be? 100ms? 1 second? Agree with the business.
💡 Key Insight

Within a single aggregate, you always have strong consistency — that's the point of the aggregate boundary. Across aggregate boundaries (especially across bounded contexts), you accept eventual consistency. This is a deliberate design decision, not a compromise. The aggregate IS the consistency boundary.

✅ Rule of Thumb

Ask: "Does the user need to see this change reflected immediately in the same HTTP response?" If yes → same aggregate, same transaction (strong consistency). If no → event-driven, eventual consistency is fine.

Chapter 26Aggregate Design Heuristics

Vaughn Vernon's four rules for designing aggregates that scale and stay consistent.

Beyond "keep aggregates small", Vaughn Vernon (Implementing Domain-Driven Design) defines four specific rules. These are the most practically useful guidelines for aggregate design.

Rule 1: Model True Invariants in a Single Aggregate

An invariant is a business rule that must always be true. The aggregate boundary should be drawn around the objects that must be consistent with each other to enforce that invariant.

// The invariant: "balance can never go negative"
// To enforce this, you only need: current balance
// You do NOT need: transaction history, customer details, account settings
// Therefore: BankAccount aggregate contains ONLY balance (+ lock status + version)

class BankAccount {
  private _balance: Money;     // ← needed to check the invariant
  private _isLocked: boolean;  // ← needed for a different invariant

  withdraw(amount: Money): void {
    // ✅ Can enforce "no negative balance" with just _balance
    if (!this._balance.isGreaterThanOrEqual(amount)) throw new InsufficientFundsError(...);
    this._balance = this._balance.subtract(amount);
  }
}

Rule 2: Reference Other Aggregates by ID Only

Aggregates should never hold direct object references to other aggregates. Only reference by ID (a Value Object). This prevents loading too much data and enforces the boundary.

// ❌ Wrong: holding a direct reference to another aggregate
class Order {
  private customer: Customer;   // loads entire Customer aggregate every time Order loads
  private items: Product[];     // loads entire Product aggregates too
}

// ✅ Correct: reference by ID only
class Order {
  private customerId: CustomerId;   // just the ID — lightweight
  private lines: OrderLine[];       // OrderLine is an internal entity (owned by Order)
}

// When you need customer details, load them separately in the use case:
class PlaceOrderUseCase {
  async execute(command: PlaceOrderCommand) {
    const order    = await this.orderRepo.findById(command.orderId);
    const customer = await this.customerRepo.findById(order.customerId);  // load separately
    // ... use both
  }
}

Rule 3: One Aggregate Modified Per Transaction

A single business operation should modify only one aggregate. If you think you need to update two aggregates atomically, it usually means your aggregate boundaries are wrong — OR you should accept eventual consistency between them.

// ❌ Trying to update two aggregates in one transaction — design smell
async transfer(fromId: AccountId, toId: AccountId, amount: Money) {
  const from = await this.accountRepo.findById(fromId);
  const to   = await this.accountRepo.findById(toId);
  from.withdraw(amount);
  to.deposit(amount);
  await this.accountRepo.save(from);   // transaction 1
  await this.accountRepo.save(to);     // transaction 2 — if this fails, inconsistent!
}

// ✅ Option A: Use a Transfer aggregate that owns the process
class MoneyTransfer {
  private fromAccountId: AccountId;
  private toAccountId: AccountId;
  private amount: Money;
  private status: TransferStatus;   // PENDING → COMPLETED
}
// Each account updates itself when it receives the relevant event

// ✅ Option B: Accept eventual consistency via domain events
// from.withdraw() → raises MoneyWithdrawnEvent
// Event handler: to.deposit(amount) — separate transaction, eventual consistency

Rule 4: Use Eventual Consistency Between Aggregates

When an operation touches multiple aggregates, use domain events and eventual consistency rather than a distributed transaction. This is the combination of Rule 3 with the event-driven patterns you've already learned.

✅ All Four Rules Together

1. True invariants in one aggregate — only include what's needed to enforce business rules
2. Reference by ID only — no direct object refs to other aggregates
3. One aggregate per transaction — don't save two aggregates in one DB transaction
4. Eventual consistency between aggregates — use events when multiple aggregates need to react

Follow these four rules and your aggregates will be small, fast, testable, and scalable.

Appendix BDDD Terminology Glossary

Formal definitions for DDD-specific terms used throughout this guide or in the DDD community.

Rich Domain Model
Domain objects that contain both data and behaviour. Methods on aggregates and entities enforce business rules, raise events, and express domain concepts — they're not just passive data containers. The goal of DDD tactical design.
Contrast with Anemic Domain Model — the anti-pattern where objects are just structs and all logic lives in services.
Anemic Domain Model anti-pattern
A domain model where classes are nothing but data with getters/setters, and all business logic lives in service classes. Coined by Martin Fowler as an anti-pattern. The services become procedural scripts, the "domain" objects carry no intelligence.
Sign you have one: your aggregate has 20 fields, 0 business methods, and a 500-line service that operates on it.
Invariant
A business rule that must always be true within an aggregate — before any operation, during, and after. The aggregate is responsible for enforcing all its invariants. If an operation would violate an invariant, the aggregate throws a domain error and refuses.
Examples: "BankAccount balance can never be negative", "Order can't have zero items when placed", "confirmed Order can't be cancelled".
Consistency Boundary
The boundary within which all invariants must be enforced atomically. The aggregate IS the consistency boundary — everything inside is always consistent. Across aggregate boundaries, you accept eventual consistency. You can't have strong consistency across two separate aggregates without a distributed transaction (which you avoid).
Rule: one aggregate = one transaction = one consistency boundary.
Domain Expert
A person with deep knowledge of the business domain — not a developer. They understand the rules, the edge cases, the vocabulary, and the "why" behind business decisions. In DDD, domain experts collaborate directly with developers to build the domain model. The ubiquitous language is built with them, not derived without them.
Without domain experts, you model your assumptions about the business — not the business itself.
Domain Primitive also: Tiny Type
A Value Object that wraps a single primitive type to give it domain meaning and safety. Instead of passing raw string IDs around, you use AccountId. The compiler then prevents you from accidentally passing a TransactionId where an AccountId is expected.
// Without Domain Primitive — compiler can't help you
function findAccount(id: string) { ... }
findAccount(transactionId);   // compiles — wrong!

// With Domain Primitive — caught at compile time
function findAccount(id: AccountId) { ... }
findAccount(transactionId);   // ❌ Type error — caught!
Projection also: Read Model, View Model
In CQRS, a projection is a denormalised, query-optimised view of data built by processing domain events. Unlike aggregates (optimised for enforcing rules), projections are optimised for fast, specific reads. They're maintained by event handlers that react to domain/integration events and update dedicated read tables.
// Projection updated by event handler
@OnEvent('MoneyWithdrawnDomainEvent')
async updateAccountSummaryProjection(event: MoneyWithdrawnDomainEvent) {
  await this.db.query(`
    UPDATE account_summary_projection
    SET balance = balance - $1, last_transaction_at = NOW()
    WHERE account_id = $2
  `, [event.amount, event.accountId]);
}
Transaction Script alternative pattern
A simpler alternative to Domain Model. Each business operation is a single procedural function (script) that works directly with database rows. No aggregates, no domain objects — just a function that does the work from start to finish. Fast to write, appropriate for simple or low-complexity features.
Use for Supporting/Generic subdomains. Use Domain Model (DDD) for Core subdomains. The cost of DDD only pays off where complexity warrants it.
// Transaction Script — fine for simple features
async function updateUserPreferences(userId: string, prefs: PrefsDto) {
  await db.query('UPDATE users SET prefs = $1 WHERE id = $2', [prefs, userId]);
}
Published Language
A Context Map relationship pattern where a bounded context publishes a well-documented, versioned data format that other contexts use to integrate. The publishing context owns and maintains the schema. Consumers conform to it. Often realised as event schemas (Avro, JSON Schema, Protobuf) or an API spec (OpenAPI).
Example: BankAccount context publishes a Kafka topic with a versioned Avro schema for MoneyWithdrawnIntegrationEvent. All consuming contexts conform to that schema.
Model-Driven Design
The DDD principle that the software model should directly and faithfully reflect the domain model. Code and domain model are the same thing — not a translation of it. When the domain model changes (domain experts change their understanding), the code changes in the same way. There's no separate "analysis model" and "implementation model".
The alternative (translation) is where concepts exist in requirements but are absent in code — then bugs hide in the translation gap.
Big Ball of Mud anti-pattern
A system with no clear structure, boundaries, or architecture. Everything is tangled with everything else. Common in long-lived codebases where design was never deliberately enforced. Recognised by Brian Foote and Joseph Yoder as the most common real-world "architecture." DDD's bounded contexts and layered architecture are the structured alternative.
Symptoms: any class can import any other class, no module/context boundaries, changing one thing breaks unrelated things, nobody fully understands the system.
Distillation
Eric Evans' concept of progressively separating the core domain from everything that supports it. You identify the essential concepts that make your product unique and isolate them from generic mechanisms and supporting logic. The goal: the core domain code should be clean, focused, and expressive — unburdened by technical infrastructure or generic concerns.
Practical application: when you apply Subdomains (Chapter 3) and focus DDD effort only on Core, you're practising distillation.

Chapter 29Testing Strategy in DDD

Which layer you test, how, and what never to mock.

DDD's layered architecture maps cleanly onto a testing pyramid. Each layer has a natural test type — the key insight is that your domain layer tests are pure unit tests with zero dependencies. No mocks, no database, no framework.

▲ E2E Tests (few) — full HTTP request → DB → response
▲▲ Integration Tests (some) — Use Case + real repository + real DB
▲▲▲▲ Unit Tests (many) — Aggregate + Value Objects, no infrastructure at all

Layer 1 — Unit Testing the Domain (Aggregates & Value Objects)

The domain layer has no dependencies — no database, no HTTP, no NestJS. Tests are fast, pure, and require zero mocking. This is the highest-value test layer in DDD because it directly tests your business rules.

// bank-account.aggregate.spec.ts
// ✅ No imports from infrastructure, no mocks, no database
describe('BankAccount', () => {
  describe('withdraw', () => {
    it('reduces balance by the withdrawn amount', () => {
      const account = BankAccount.open('USD');
      account.deposit(Money.of(100, 'USD'));

      account.withdraw(Money.of(30, 'USD'));

      expect(account.balance.amount).toBe(70);
    });

    it('throws InsufficientFundsError when balance is too low', () => {
      const account = BankAccount.open('USD');
      // balance is 0

      expect(() => account.withdraw(Money.of(50, 'USD')))
        .toThrow(InsufficientFundsError);
    });

    it('throws AccountLockedError when account is locked', () => {
      const account = BankAccount.open('USD');
      account.deposit(Money.of(100, 'USD'));
      account.lock();

      expect(() => account.withdraw(Money.of(10, 'USD')))
        .toThrow(AccountLockedError);
    });

    it('raises MoneyWithdrawnEvent after successful withdrawal', () => {
      const account = BankAccount.open('USD');
      account.deposit(Money.of(100, 'USD'));

      account.withdraw(Money.of(40, 'USD'));

      const events = account.pullDomainEvents();
      // open() raises AccountOpenedEvent, deposit() raises MoneyDepositedEvent,
      // so MoneyWithdrawnEvent is the last one
      const withdrawnEvent = events.find(e => e instanceof MoneyWithdrawnEvent);
      expect(withdrawnEvent).toBeDefined();
      expect((withdrawnEvent as MoneyWithdrawnEvent).amount).toBe(40);
      expect((withdrawnEvent as MoneyWithdrawnEvent).newBalance).toBe(60);
    });
  });
});

// money.vo.spec.ts — Value Objects tested the same way
describe('Money', () => {
  it('throws when amount is negative', () => {
    expect(() => Money.of(-10, 'USD')).toThrow();
  });

  it('two Money instances with same amount and currency are equal', () => {
    expect(Money.of(100, 'USD').equals(Money.of(100, 'USD'))).toBe(true);
  });
});

Layer 2 — Integration Testing Use Cases

Use cases orchestrate the domain + repository + events. Test them with a real (in-memory or test) database — not mocked repositories. This is the most important rule: do not mock your repository in use case tests.

⚠️ Why Not to Mock the Repository

If you mock the repository, your test never exercises the real persistence logic. When you ship, you may discover that the ORM mapping is wrong, the version column isn't updating, or the reconstitution logic has a bug. Mocked repository tests are false confidence. Use a real SQLite in-memory database instead — it's fast enough.

// deposit-money.use-case.integration.spec.ts
describe('DepositMoneyUseCase (integration)', () => {
  let useCase: DepositMoneyUseCase;
  let accountRepo: IBankAccountRepository;
  let dataSource: DataSource;

  beforeAll(async () => {
    // Spin up a real in-memory SQLite DB — same TypeORM config, no mocks
    dataSource = await createTestDataSource(); // sqlite :memory:
    accountRepo = new BankAccountTypeOrmRepository(dataSource.getRepository(BankAccountOrmEntity));
    useCase = new DepositMoneyUseCase(accountRepo, new TransactionTypeOrmRepository(...), new EventEmitter2());
  });

  afterAll(() => dataSource.destroy());

  it('persists the updated balance after deposit', async () => {
    // Arrange: save an account to the real DB
    const openResult = await new OpenAccountUseCase(accountRepo, new EventEmitter2()).execute({ currency: 'USD' });

    // Act: deposit via the use case
    const result = await useCase.execute({
      accountId: openResult.accountId,
      amount: 500,
      currency: 'USD',
      description: 'Salary',
    });

    // Assert: re-load from DB and confirm balance is actually saved
    const reloaded = await accountRepo.findById(AccountId.of(openResult.accountId));
    expect(reloaded!.balance.amount).toBe(500);
    expect(result.newBalance).toBe(500);
  });

  it('increments the version after save (optimistic locking)', async () => {
    const openResult = await new OpenAccountUseCase(accountRepo, new EventEmitter2()).execute({ currency: 'USD' });
    const account = await accountRepo.findById(AccountId.of(openResult.accountId));
    expect(account!.version).toBe(0); // fresh account

    await useCase.execute({ accountId: openResult.accountId, amount: 100, currency: 'USD', description: 'test' });

    const reloaded = await accountRepo.findById(AccountId.of(openResult.accountId));
    expect(reloaded!.version).toBe(1); // incremented after save
  });
});

Chapter 28Q & A — Deep Questions

The questions developers actually ask when applying DDD in real projects.

1If bounded contexts can't directly access each other's data, how does one context get data from another?

This is the most common real-world friction point. You have three options, and the community has a clear preference order:

Option A — Synchronous API call (Query) — when you need data right now to complete a request. Context A makes an HTTP/gRPC call to Context B's public endpoint. Context B exposes only the data the outside world needs — not its full internal model.

// KYC context needs customer name from Auth context (synchronous need)
@Injectable()
export class KycService {
  async startVerification(customerId: string): Promise {
    // Call Auth context's public API — goes through ACL to translate
    const customerInfo = await this.authContextClient.getCustomerBasicInfo(customerId);
    // customerInfo is Auth context's public DTO, NOT Auth's domain model
    await this.kycRepo.save(KycCase.open(customerId, customerInfo.legalName));
  }
}

Option B — Integration Events (Async) — when you need to react to something that happened, or when the data can be stale for a short time. Context A publishes an event; Context B subscribes and stores a local copy of what it needs.

// Payments context stores the customer name it needs locally,
// updated whenever Auth context fires a CustomerRegisteredEvent
@Injectable()
export class CustomerRegisteredHandler {
  @OnEvent('CustomerRegisteredIntegrationEvent')
  async handle(event: CustomerRegisteredIntegrationEvent) {
    // Payments context stores ONLY what it needs — its own projection
    await this.billingProfileRepo.save(BillingProfile.create({
      customerId: event.customerId,
      name: event.name,     // local copy — payments context owns this now
      currency: event.preferredCurrency,
    }));
  }
}
// Now Payments context never calls Auth context at runtime —
// it has its own copy. This is called "data denormalization across contexts".

Option C — Shared Read Model (Same DB only) — if both contexts live in the same database (monolith), Context B can have a read-only query on Context A's tables. Never write to another context's tables. This is a pragmatic monolith compromise — avoid in microservices.

✅ Community Approach

Prefer Integration Events + local copy (Option B) for data that changes infrequently and where eventual consistency is acceptable. Use synchronous API calls (Option A) only when you need real-time data to complete a request. Never have Context A reach into Context B's database directly.

2Should each bounded context have its own database, or can they share one?

Ideal (microservices): Each bounded context has its own database — different technology even. Auth uses Postgres, Search uses Elasticsearch, Sessions uses Redis. This gives full isolation but requires integration events for communication.

Pragmatic (modular monolith): Same database, but separate schemas or at least strict table ownership. Context A never queries Context B's tables except through Context B's code.

// Same Postgres DB, but separate schemas — good compromise
// auth schema: auth.customers, auth.sessions
// payments schema: payments.billing_profiles, payments.transactions
// kyc schema: kyc.verifications

// Each module connects to its own schema:
TypeOrmModule.forRoot({ schema: 'payments', entities: [...paymentsEntities] })
TypeOrmModule.forRoot({ schema: 'auth',     entities: [...authEntities] })
✅ Community Approach

Start with a modular monolith with separate schemas. The schema boundary enforces context separation without the operational overhead of multiple databases. When a context needs to scale independently or the team grows, extract it to a separate service with its own DB. Don't start with microservices.

3Where does validation live — DTO, Use Case, or Aggregate? I'm confused about who validates what.

Three different kinds of validation, three different places:

Validation TypeWhereExample
Format / type validationDTO (Presentation layer)"amount must be a positive number", "currency must be 3 chars"
Business rule validationAggregate (Domain layer)"balance can't go negative", "can't withdraw from locked account"
Cross-aggregate validationDomain Service or Use Case"customer can't open more than 3 accounts"
 
// 1. DTO — format validation (NestJS class-validator)
export class WithdrawDto {
  @IsUUID() accountId: string;
  @IsPositive() @IsNumber() amount: number;
  @Length(3, 3) currency: string;
  @IsNotEmpty() description: string;
}

// 2. Value Object — self-validates on construction
class Money {
  private constructor(amount: number, currency: string) {
    if (amount < 0) throw new Error('negative money');       // ← format/domain rule in VO
    if (currency.length !== 3) throw new Error('bad currency');
  }
}

// 3. Aggregate — business rule validation
class BankAccount {
  withdraw(amount: Money): void {
    if (this._isLocked) throw new AccountLockedError();          // ← business rule
    if (!this._balance.isGreaterThanOrEqual(amount)) throw new InsufficientFundsError(); // ← business rule
  }
}

// 4. Use Case — cross-aggregate (load and check)
class WithdrawMoneyUseCase {
  async execute(cmd) {
    const account = await this.repo.findById(AccountId.of(cmd.accountId));
    if (!account) throw new AccountNotFoundError();   // ← existence check, not in aggregate
    account.withdraw(Money.of(cmd.amount, cmd.currency));  // aggregate does business validation
  }
}
✅ Rule of Thumb

If a rule is about format, validate in the DTO. If a rule is about a business invariant, validate in the aggregate (or Value Object). If a rule requires loading other data, validate in the Use Case or Domain Service. The Use Case should never contain pure business logic — it orchestrates, not decides.

4Can an aggregate call a repository internally to load related data?

No. Aggregates must be pure domain objects. They receive everything they need through their constructor or method parameters. They never load data themselves.

If an aggregate could call a repository, it would need to depend on infrastructure, defeating the entire purpose of the clean architecture. It also makes aggregates impossible to test without a database.

// ❌ Wrong — aggregate calls repository
class Order {
  addItem(productId: ProductId): void {
    const product = this.productRepo.findById(productId); // ← NO. Aggregate can't do this.
    this._items.push(new OrderItem(productId, product.price));
  }
}

// ✅ Correct — use case loads everything needed, passes to aggregate
class AddItemToOrderUseCase {
  async execute(cmd: AddItemCommand): Promise {
    const order   = await this.orderRepo.findById(OrderId.of(cmd.orderId));
    const product = await this.productRepo.findById(ProductId.of(cmd.productId));
    // Pass the data the aggregate needs — it doesn't load anything itself
    order.addItem(product.id, product.price, product.name);
    await this.orderRepo.save(order);
  }
}
💡 Key Insight

The use case is the orchestrator — it knows what data to load and in what order. The aggregate is the rule-enforcer — it only receives what it needs and decides if the operation is valid. Never blur these responsibilities.

5When should I put logic in a Domain Service vs directly on the Aggregate?

Simple decision tree:

  • Does the operation involve only one aggregate? → Put it on the aggregate as a method.
  • Does the operation involve two or more aggregates? → Domain Service.
  • Does the operation require loading data (repositories)? → Use Case (not domain service).
  • Does the concept feel awkward as a method on either aggregate ("it doesn't really belong here")? → Domain Service.
// ✅ On the aggregate — only involves BankAccount
account.withdraw(amount);      // single aggregate → aggregate method
account.lock();
account.deposit(amount);

// ✅ Domain Service — naturally involves TWO aggregates
class MoneyTransferService {
  // "transfer" doesn't belong to either account — it's a concept between them
  transfer(from: BankAccount, to: BankAccount, amount: Money): void {
    from.withdraw(amount);
    to.deposit(amount);
  }
}

// ✅ Use Case — needs to LOAD aggregates, not just operate on them
class TransferMoneyUseCase {
  async execute(cmd) {
    const from = await this.repo.findById(AccountId.of(cmd.fromId));
    const to   = await this.repo.findById(AccountId.of(cmd.toId));
    this.transferService.transfer(from, to, Money.of(cmd.amount, cmd.currency));
    await this.repo.save(from);
    await this.repo.save(to);
  }
}
⚠️ Common Mistake

Creating Domain Services for everything and leaving aggregates as data bags (anemic domain model). Before reaching for a Domain Service, ask: "Can this naturally live on one of the aggregates?" — most of the time the answer is yes.

6How does pagination work? Should my repository load a list of aggregates?

This is where CQRS pays off. Repositories are for loading single aggregates by ID. Paginated lists are a query concern, not a domain concern — they bypass the aggregate entirely.

// ❌ Wrong — repository loading a list of aggregates for display
class BankAccountRepository {
  findAll(page: number, limit: number): Promise { ... }
  // Loading 20 full aggregates just to show a list of names + balances is wasteful
}

// ✅ Correct — CQRS read side: direct query, returns a flat DTO (no aggregate)
@Injectable()
export class ListAccountsQueryHandler {
  constructor(@InjectDataSource() private db: DataSource) {}

  async execute(query: ListAccountsQuery): Promise {
    return this.db.query(`
      SELECT id, balance_amount, balance_currency, is_locked, created_at
      FROM bank_accounts
      WHERE customer_id = $1
      ORDER BY created_at DESC
      LIMIT $2 OFFSET $3
    `, [query.customerId, query.limit, query.offset]);
    // Returns flat DTOs, not aggregates. Fast. Paginated.
  }
}
✅ Rule of Thumb

Repository = load one aggregate by ID (for write operations). Query Handler = direct DB query returning a flat DTO (for read operations / lists / pagination). Don't use aggregates for display purposes.

7My aggregate is growing large and complex. What do I do?

A growing aggregate is a signal that it's trying to do too much. You have two tools:

Tool 1 — Apply the bounded aggregate principle. Remove anything from the aggregate that isn't needed to enforce its core invariants. Move transaction history, audit logs, or display data to separate entities with their own repositories.

Tool 2 — Split into multiple aggregates communicating via events. If the aggregate has multiple distinct responsibilities, separate them.

// ❌ God Aggregate — one aggregate doing too much
class CustomerAccount {
  private _balance: Money;
  private _transactions: Transaction[];      // 1000s of rows
  private _kycStatus: KycStatus;            // belongs to KYC context
  private _notificationPreferences: Prefs;  // belongs to notifications
  private _loginHistory: LoginEvent[];       // belongs to auth
  private _investmentPortfolio: Portfolio;  // separate domain entirely
}

// ✅ Split — each aggregate owns only its invariants
class BankAccount {
  private _balance: Money;       // invariant: can't go negative
  private _isLocked: boolean;    // invariant: no transactions when locked
  // That's it. Everything else is separate.
}
// Transactions → separate entity, separate repository
// KYC → separate bounded context, separate aggregate
// Notifications → separate bounded context
// Portfolio → separate aggregate in investments context

// BankAccount raises events → other contexts react
account.withdraw(amount);
// raises MoneyWithdrawnEvent → KYC context can flag it
//                            → Notifications context sends email
//                            → Portfolio context adjusts cash position
8Can bounded contexts share Value Objects like Money or Address?

This is a nuanced question. The short answer: technically yes, but be careful.

If Money in the Payments context and Money in the Reporting context truly mean the same thing with the same rules — sharing is fine via a Shared Kernel. But they often diverge over time: Reporting Money might need currency conversion logic, Payments Money might need precision down to 8 decimal places for crypto.

// Option A: Shared Kernel — literal code sharing (risky, tight coupling)
// Both contexts import from a shared library:
import { Money } from '@moonfare/shared-domain';

// Option B: Each context has its own Money (recommended)
// payments/domain/value-objects/money.vo.ts
// reporting/domain/value-objects/money.vo.ts
// They look the same today but can evolve independently.
// When a Payments Money change would break Reporting, you'll be glad they're separate.

// The bridge: Integration Events use primitives (no shared domain objects)
class MoneyWithdrawnIntegrationEvent {
  constructor(
    readonly amount: number,    // primitive — not Money VO
    readonly currency: string,  // primitive — not Money VO
  ) {}
  // Each consuming context reconstructs its own Money from these primitives
}
✅ Community Approach

Copy Value Objects across contexts rather than share them. Duplication is cheaper than coupling. Two contexts having their own Money class is fine — they're small, and keeping them independent prevents one context's evolving rules from breaking another.

9How does authentication and authorization work in DDD? Where does it live?

Auth is a Generic subdomain (or Supporting). It doesn't belong in your core domain aggregates. The pattern:

  • Authentication (who are you?) — a separate bounded context. Use Auth0, Cognito, or build your own Auth context. Your domain aggregates never know about JWT tokens.
  • Authorization (are you allowed?) — NestJS Guards handle this at the presentation layer, before the use case is called. The use case receives the authenticated user's ID as a plain value.
// Guard resolves identity from JWT — presentation layer concern
@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {}

// Controller: authenticated userId is extracted before reaching use case
@Post(':id/withdraw')
@UseGuards(JwtAuthGuard)
async withdraw(
  @CurrentUser() userId: string,   // resolved by guard from JWT
  @Param('id') accountId: string,
  @Body() body: WithdrawDto,
) {
  return this.withdrawUseCase.execute({
    requestingUserId: userId,   // passed into use case as a plain string
    accountId,
    amount: body.amount,
    currency: body.currency,
  });
}

// Use Case: checks ownership as a business rule — not a security concern
class WithdrawMoneyUseCase {
  async execute(cmd: WithdrawMoneyCommand) {
    const account = await this.repo.findById(AccountId.of(cmd.accountId));
    // Business rule: you can only withdraw from your own account
    if (account.ownerId.value !== cmd.requestingUserId) {
      throw new UnauthorizedAccountAccessError();
    }
    account.withdraw(Money.of(cmd.amount, cmd.currency));
  }
}
// Domain aggregate never imports JWT, Guards, or auth libraries
10What's the difference between a Command and an Event? When do I use each?

Fundamental distinction — they flow in opposite directions and have different semantics:

CommandEvent
RepresentsAn intent — "please do this"A fact — "this already happened"
Can be rejected?Yes — validation may failNo — it already happened, can't un-happen
TenseImperative: WithdrawMoneyPast tense: MoneyWithdrawn
TargetOne handler (one use case)Zero or many handlers
DirectionCaller → systemSystem → subscribers
 
// COMMAND: intent, directed at one use case, can fail
export class WithdrawMoneyCommand {
  constructor(
    readonly accountId: string,
    readonly amount: number,
    readonly currency: string,
  ) {}
}
// Sent to exactly ONE handler. If the account is locked, command is rejected.

// EVENT: fact, broadcast to any listeners, cannot be rejected
export class MoneyWithdrawnEvent {
  constructor(
    readonly accountId: string,
    readonly amount: number,
    readonly occurredAt: Date,
  ) {}
}
// Broadcast to ZERO or MANY handlers. It already happened — nobody can reject it.
// FraudHandler, EmailHandler, AuditHandler all react independently.
💡 Key Insight

Events describe the past — they're immutable facts. Commands describe the future — they express intent. This is why event names are past tense and command names are imperative. Getting this right in your naming makes the codebase self-documenting.

11Can I use DDD in a monolith, or is it only for microservices?

DDD is completely independent of deployment architecture. In fact, Eric Evans' original book describes a monolith. DDD is about how you model and structure your code — not how you deploy it.

A modular monolith is often the best starting point: bounded contexts as NestJS modules in a single deployable app, communicating via in-process events. When a context needs to scale independently, extract it to a microservice — the bounded context boundary is already there.

// Modular monolith: all contexts in one app
// NestJS modules enforce bounded context boundaries
@Module({
  imports: [
    BankAccountModule,    // bounded context 1
    KycModule,            // bounded context 2
    NotificationsModule,  // bounded context 3
    AuthModule,           // bounded context 4
  ],
})
export class AppModule {}

// Contexts communicate via in-process events (NestJS EventEmitter2)
// When ready to extract KycModule to its own service:
// 1. Replace EventEmitter2 with Kafka publisher/consumer
// 2. Deploy as separate service
// 3. The domain code inside KycModule doesn't change at all
✅ Community Approach

Start as a modular monolith. Get the bounded context boundaries right first. Extract to microservices only when you have a specific reason (independent scaling, team ownership, different deployment frequency). The #1 microservices mistake is doing it too early, before the boundaries are understood.

12How do I migrate an existing spaghetti NestJS app to DDD? Do I rewrite?

Never rewrite from scratch — the "big bang rewrite" is a well-documented disaster pattern. Migrate incrementally using the Strangler Fig pattern.

  1. Identify bounded contexts — run Event Storming or just map the major domain areas. Don't need to be perfect.
  2. Start with one context — pick the most complex or most painful one (your core subdomain).
  3. Extract the domain model — create the aggregate, value objects, and repository interface. Leave the existing service/controller in place.
  4. Introduce the repository — the existing TypeORM entities become the ORM entities in the infrastructure layer. Map them to the new aggregate.
  5. Route new features through the new model — don't touch old code. New features go through the DDD path.
  6. Gradually migrate old code — as you touch old methods for bugs/changes, migrate them to the new model.
// Phase 1: new aggregate exists alongside old service
// Old code still works:
class AccountService {
  async withdraw(accountId: string, amount: number) {
    const account = await this.accountRepo.findOne(accountId);
    if (account.balance < amount) throw new Error('Insufficient funds');
    account.balance -= amount;
    await this.accountRepo.save(account);
  }
}

// New DDD path for new features — runs parallel
class WithdrawMoneyUseCase {
  async execute(cmd: WithdrawMoneyCommand) {
    const account = await this.bankAccountRepo.findById(AccountId.of(cmd.accountId));
    account.withdraw(Money.of(cmd.amount, cmd.currency));
    await this.bankAccountRepo.save(account);
  }
}
// Both read/write the same DB table during migration — that's fine.
// Gradually retire AccountService methods as you migrate them.
✅ Rule of Thumb

Migrate towards DDD, not to DDD all at once. One context, one aggregate, one use case at a time. A half-DDD codebase is better than a broken rewrite.

13How do I handle reporting and analytics? Should aggregates support complex queries?

No. Reporting is almost always a separate bounded context (or at minimum a separate read model). Aggregates are designed for enforcing rules — not for complex joins, date range queries, or aggregations.

// ❌ Wrong — domain aggregate trying to support reporting
class BankAccount {
  getTransactionSummaryByMonth(year: number, month: number) { ... }
  calculateAverageBalance(from: Date, to: Date) { ... }
  // Aggregates don't do this
}

// ✅ Option A — CQRS read model: query handler with direct SQL
@Injectable()
export class MonthlyStatementQueryHandler {
  async execute(query: MonthlyStatementQuery): Promise {
    return this.db.query(`
      SELECT
        DATE_TRUNC('month', created_at) as month,
        SUM(CASE WHEN type='DEPOSIT' THEN amount ELSE 0 END) as total_deposited,
        SUM(CASE WHEN type='WITHDRAWAL' THEN amount ELSE 0 END) as total_withdrawn,
        COUNT(*) as transaction_count
      FROM transactions
      WHERE account_id = $1 AND created_at BETWEEN $2 AND $3
      GROUP BY 1
    `, [query.accountId, query.from, query.to]);
  }
}

// ✅ Option B — separate Reporting bounded context
// Events flow from BankAccount context → Reporting context maintains its own
// denormalised tables optimised purely for reporting queries.
// Reporting context doesn't need aggregates at all — just projections.
💡 Key Insight

The separation between "write model" (aggregates) and "read model" (projections/reporting) is one of the biggest practical wins of DDD + CQRS. Your aggregates stay small and fast. Your reports can be as complex as needed without touching domain logic.

14How do I test that my aggregate raised the right domain events?

Because aggregates collect events internally (via pullDomainEvents()), testing them is straightforward — no mocks needed.

describe('BankAccount domain events', () => {
  it('should raise MoneyWithdrawnEvent after a successful withdrawal', () => {
    // Arrange
    const account = BankAccount.open('USD');
    account.deposit(Money.usd(500));
    account.pullDomainEvents(); // clear the deposit event

    // Act
    account.withdraw(Money.usd(200));

    // Assert
    const events = account.pullDomainEvents();
    expect(events).toHaveLength(1);
    expect(events[0]).toBeInstanceOf(MoneyWithdrawnEvent);

    const event = events[0] as MoneyWithdrawnEvent;
    expect(event.amount).toBe(200);
    expect(event.currency).toBe('USD');
    expect(event.accountId).toBe(account.id.value);
  });

  it('should NOT raise events if withdrawal fails', () => {
    const account = BankAccount.open('USD');
    account.deposit(Money.usd(100));
    account.pullDomainEvents();

    // Act: attempt overdraft
    expect(() => account.withdraw(Money.usd(500))).toThrow(InsufficientFundsError);

    // No events — nothing happened
    const events = account.pullDomainEvents();
    expect(events).toHaveLength(0);
  });
});
✅ Rule of Thumb

Testing domain events should be pure unit tests — no NestJS, no EventEmitter2, no DB. Just instantiate the aggregate, call the method, and call pullDomainEvents() to inspect what was raised. Fast, deterministic, no infrastructure.

15When should I NOT apply DDD patterns? Is there such a thing as over-engineering with DDD?

Absolutely. DDD introduces real complexity — layers, abstractions, interfaces, mappers. That complexity only pays off when the domain itself is complex. Applying full DDD to a simple feature is over-engineering.

ScenarioAppropriate Approach
Simple CRUD (create user, list users, update profile)Transaction Script — controller → service → TypeORM entity. No aggregate needed.
Admin panel — mostly data entry and displaySimple CRUD with NestJS + TypeORM. DDD would be overkill.
Complex business rules (financial calculations, investment rules, compliance)Full DDD — aggregates, domain services, events.
Reporting / analyticsCQRS read model or separate reporting service. No aggregates.
File processing pipelinePipeline pattern — no aggregates needed.

Even within a DDD codebase, not every feature needs a full aggregate. A simple user preferences feature can be a straightforward TypeORM entity saved by a service — no aggregate, no domain events, no value objects. Save the ceremony for where the business complexity warrants it.

💡 Key Insight

DDD is a strategic investment. The cost is real (more code, more layers). The ROI only materialises in genuinely complex domains with rich business rules that change frequently. Eric Evans himself says: don't apply DDD where the domain is simple. Use the simplest design that works.

16Two aggregates need to be updated together or the data is inconsistent. Do I use a distributed transaction?

No. Distributed transactions (two-phase commit) are fragile, slow, and don't scale. The DDD community consensus is to avoid them entirely. Instead, accept eventual consistency using the Saga pattern or domain events.

// ❌ Trying to save two aggregates atomically — fragile
await this.accountRepo.save(fromAccount);   // succeeds
await this.accountRepo.save(toAccount);     // fails → from has been debited, to hasn't been credited → inconsistent

// ✅ Option A: Saga with compensation
class TransferMoneySaga {
  async execute(cmd: TransferMoneyCommand): Promise {
    try {
      await this.withdrawUseCase.execute({ accountId: cmd.fromId, amount: cmd.amount });
      // withdraw succeeded — fromAccount debited
      try {
        await this.depositUseCase.execute({ accountId: cmd.toId, amount: cmd.amount });
        // both succeeded ✅
      } catch (depositError) {
        // deposit failed — compensate: reverse the withdrawal
        await this.depositUseCase.execute({ accountId: cmd.fromId, amount: cmd.amount });
        throw new TransferFailedError('deposit failed, withdrawal reversed');
      }
    } catch (withdrawError) {
      throw new TransferFailedError('insufficient funds');
    }
  }
}

// ✅ Option B: Intermediate aggregate owns the process
// Create a MoneyTransfer aggregate with status PENDING → COMPLETED/FAILED
// Each account processes its side independently, driven by events
class MoneyTransfer {
  private _status: 'PENDING' | 'DEBIT_COMPLETE' | 'COMPLETED' | 'FAILED';
  // Use case saves this aggregate, raises events
  // From account handles DebitMoneyTransferEvent → raises MoneyTransferDebited
  // To account handles MoneyTransferDebited → raises MoneyTransferCompleted
}
✅ Community Approach

Accept eventual consistency between aggregates. Design compensating actions for failure cases. Use Sagas to coordinate. The system will be temporarily inconsistent for milliseconds — that's a known, designed trade-off, not a bug.

17How do I handle soft deletes in DDD? Should aggregates have an isDeleted flag?

Soft deletes in DDD are best expressed as a domain concept rather than a technical flag. Instead of isDeleted: true, model what "deleted" actually means in your domain.

// ❌ Technical flag — no domain meaning
class BankAccount {
  private _isDeleted: boolean;   // what does this even mean in the domain?
}

// ✅ Model the actual business concept
class BankAccount {
  private _status: AccountStatus;  // 'ACTIVE' | 'CLOSED' | 'SUSPENDED'

  close(reason: AccountClosureReason): void {
    if (this._balance.amount > 0) throw new Error('Cannot close account with positive balance');
    if (this._status === AccountStatus.CLOSED) throw new Error('Already closed');
    this._status = AccountStatus.CLOSED;
    this._domainEvents.push(new AccountClosedEvent(this._id.value, reason));
  }
}

// Repository filters based on status — not isDeleted
abstract class IBankAccountRepository {
  abstract findById(id: AccountId): Promise;
  abstract findActiveById(id: AccountId): Promise;  // only ACTIVE accounts
}

// In the ORM entity: keep the row, just mark the status
@Column({ type: 'enum', enum: AccountStatus, default: AccountStatus.ACTIVE })
status: AccountStatus;
✅ Rule of Thumb

Replace isDeleted with a meaningful domain status (CLOSED, ARCHIVED, DEACTIVATED). Raise a domain event when the status changes. Never physically delete rows for business entities — you lose history. For truly technical records (sessions, temp tokens), physical deletion is fine.

18In my DB schema, should tables map 1:1 to aggregates? Or can one aggregate span multiple tables?

Both are valid — the mapping depends on the aggregate design. The key rule: the repository is responsible for loading and saving the full aggregate, regardless of how many tables it touches. The domain never knows.

// Case 1: Simple aggregate → 1 table
// BankAccount aggregate maps to exactly one bank_accounts table
// Repository does: SELECT * FROM bank_accounts WHERE id = $1

// Case 2: Complex aggregate → 2+ tables (owned entities)
// Order aggregate = Order root + OrderLine entities (OrderLines owned by Order)
// Delete Order → OrderLines also deleted (deletion test: yes, same aggregate)
@Injectable()
export class OrderTypeOrmRepository extends IOrderRepository {
  async findById(id: OrderId): Promise {
    // Joins two tables to reconstruct the full aggregate
    const row = await this.ormRepo.findOne({
      where: { id: id.value },
      relations: ['lines'],  // TypeORM eager-loads OrderLine rows
    });
    if (!row) return null;
    return Order.reconstitute({
      id: row.id,
      customerId: row.customerId,
      status: row.status,
      lines: row.lines.map(l => OrderLine.reconstitute({ productId: l.productId, quantity: l.quantity, price: l.unitPrice })),
    });
  }

  async save(order: Order): Promise {
    // Saves to both orders AND order_lines tables
    await this.ormRepo.save(this.toOrmEntity(order));
    // TypeORM cascades the order_lines save via the relation
  }
}

// Case 3: Separate aggregates → separate repositories (never joined by repository)
// BankAccount and Transaction are separate aggregates
// They each have their own repository — never loaded together
💡 Key Insight

The DB schema is an infrastructure concern. The aggregate is a domain concern. They don't have to match. Your repository is the translator. A well-designed aggregate can be stored in 1 table, 3 tables, or even a document DB — the domain code never changes.
